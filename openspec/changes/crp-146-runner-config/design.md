## Context

Slice 2 of the E2E orchestrator migration builds **everything needed to launch one model session correctly, as reusable modules** ‚Äî `config/` + `runner/` (prompt builder ¬∑ guards ¬∑ `AgentRunner` port + codex adapter ¬∑ generic executor) + `interaction/` (`04_migration_plan.md ## Slice 2 ‚Äî Runner config + agent runner + guards`). It is the executable counterpart to Slice 1: Slice 1 (`crp-145`) landed the step catalog as declarative data whose `Action[]` metadata is typed but **explicitly not executed** (its D7); Slice 2 makes those actions runnable while adding no orchestrator loop.

Current state: `packages/asdlc-coordinator/src/` holds the Slice 1 core (`workspace/`, `sequencing/` with the `STEP_CATALOG` and `Action`/`ReadOnlyGuard`/`RunPredicate` types in `sequencing/step-catalog.ts`, the `Diagnostic` type in `types/`), plus the in-process reuse targets `sync/` (`repo-br-scan`, `contract-delta`, `surface-map`, `prerequisite-gaps`), `context/` builders (each returns a `ContextResult` carrying a `.text` string; read-only inputs appear as `- read_only_input: <path>` lines **inside that text**), and `readiness/` (`br-clarification`). Slice 1's catalog typing is the right seam, but its guard assignments were too broad for execution: it effectively treated almost every session as `fromContext`, while the shell actually guards only steps 6/7/8/8.x that way, uses `mustExistUnchanged` for 5/5.1, and uses a special existence-preserving guard for 7.1. Slice 2 therefore executes the catalog **and** corrects that metadata so the runner can stay data-driven. The shell parity sources are `overmind/scripts/project_mgmt/project_add_feature_e2e.sh` ‚Äî `load_model_config` (awk over `.setup/models.md`), 13 `build_*_prompt` heredocs, 13 `run_*_skill` launchers (each: `load_model_config` ‚Üí `MODEL_CMD == codex` assertion ‚Üí `require_command` ‚Üí build prompt ‚Üí optional pre-session sync ‚Üí optional `context` + `sed` read-only extraction ‚Üí `mktemp`/`cp` snapshots ‚Üí `codex -m <model> <args> <prompt>` in a `cd runtime_root` subshell ‚Üí `cmp -s` guard verification ‚Üí required-output `[[ -f ]]` checks) ‚Äî and `.setup/models.md` (the pipe-table, format unchanged). The shell e2e stays live and keeps running its own launchers until Slice 3.

Constraints: `asdlc-coordinator`'s runtime `dependencies` stays empty (zero-runtime-dependency rule; `codex` is spawned via `node:child_process`, no library); Node's built-in test runner only; `npm run verify` (Slice 0) green locally is the completion gate; no changes to skill bodies, gates, `.setup/models.md` format, or artifact formats.

## Goals / Non-Goals

**Goals:**
- A typed `config/` loader for the **unchanged** `.setup/models.md` pipe-table: per-phase `{ command, model, args[] }`, `codex`-only command validation, load problems as `Diagnostic` values (not thrown, not mid-run awk failures).
- **One** pure prompt builder over a `StepDefinition` session action + runtime bindings, replacing the 13 heredocs, keeping skill-owned final-response lines out ‚Äî with content parity proven against the 13 heredocs.
- **One** `AgentRunner` port + `CodexAgentRunner` adapter (spawn from runtime root, inherit stdio, exit-code result), keeping codex swappable behind the seam.
- **One** guard module implementing the full `readOnlyGuards` union (`fromContext` / `mustExistUnchanged` / `preserveExistence`) + `requiredOutputs` assertion with empty-list-is-legal, applied uniformly (7.1 is data, not a branch).
- **One** generic `executeStep(stepDef, bindings)` executor iterating a step's `Action[]` with no per-step branches, preserving the D7 sync/context ordering as in-process calls, reusing `sync/`/`context/`/`readiness/` unchanged.
- An `interaction/` port + TTY adapter (today's wording) so Slice 3's loop and the extension's forms implement the same seam.

**Non-Goals:**
- No orchestrator loop / `overmind run`, no feature selection, no `PhaseOutcome` flow control, no resume/restart guidance, no checkpoint commits (Slice 3).
- No scaffold-feature primitive execution beyond the executor calling a named deterministic action (the `capture/scaffold-feature` primitive lands in Slice 3); no `state/` or `git/` modules.
- No class-list session binding, no reconciliation session, no `overmind project reconcile` (Slice 4).
- No new CLI verbs and no deletions/shell edits this slice ‚Äî the shell e2e keeps its launchers and `.setup/models.md` reads until Slice 3.
- No changes to skill bodies, gates, the `.setup/models.md` format, or artifact formats.

## Decisions

### D1 ‚Äî `config/` parses the unchanged `.setup/models.md`; the format stays, the awk goes
`loadRunnerConfig(modelsPath) ‚Üí { phases: Map<phase, { command, model, args[] }>, diagnostics[] }` parses the same `phase | command | model | args...` pipe-table rows (comment lines `^\s*#` skipped, rows with `< 3` fields ignored, fields trimmed) that `load_model_config`'s awk did (decision 5 in `03_target_architecture.md ## Decisions`). `command` is validated against the registered agent adapters ‚Äî only `codex` initially ‚Äî replacing the thirteen scattered `MODEL_CMD == codex` assertions with **one** rule. *Why:* zero migration for existing workspaces; "typed" lives in the loader and schema, not a new file format. *Errors-as-values (D4 of `crp-145`, `03_target_architecture.md ## Diagnostics`):* a missing file, a phase absent from the table, a malformed/short row, or a non-`codex` command yields a `Diagnostic` in the typed result ‚Äî actionable at startup ‚Äî never a throw and never a mid-run failure.

### D2 ‚Äî One pure prompt builder over the session action, parity-tested against the 13 heredocs
`buildSessionPrompt(sessionAction, bindings) ‚Üí string` is a single pure function replacing the 13 `build_*_prompt` heredocs (`02_responsibility_translation_map.md` row 7). Bindings carry the runtime root, feature path, overmind CLI path, target class (step 7), and the resolved artifact paths; the builder emits the same three-part shape the heredocs did ‚Äî `Use/Load the overmind-<skill> skill`, a `Runtime bindings:` block, and a `Required flow:` block with the **exact** `node <cli> capture|context|sync|gate <skill> <feature> [--class <c>]` command lines ‚Äî and **omits** skill-owned final-response lines exactly as today. *Why:* the 13 heredocs are near-identical modulo skill name, bindings, and command set; the differences are already data on the `StepDefinition` session action (`skillName`, `modelPhase`, `requiresSync`) plus the resolved bindings. *Parity bar:* a test asserts, per phase, the skill name, the runtime-binding lines, the exact CLI command lines, and the absence of final-response lines ‚Äî the "prompt content parity against the 13 heredocs" acceptance criterion.

### D3 ‚Äî `AgentRunner` port + `CodexAgentRunner` adapter; stdio inherited, no fd dance
`interface AgentRunner { run(spec: { command, model, args[], prompt, cwd }) ‚Üí { exitCode } }`; `CodexAgentRunner` spawns `codex -m <model> <args...> <prompt>` from the runtime root with **inherited stdio** via `node:child_process` (`02_responsibility_translation_map.md` row 8; `03_target_architecture.md ## Runner`). Inherited stdio makes interactive sessions the default ‚Äî the shell's fd-9 forwarding dance was a bash artifact with no TS equivalent. *Why:* the port keeps a second runner (Claude, etc.) a pure add-on; config's `codex`-only rule (D1) is the current guard, lifted when a second adapter is actually wanted (`03_target_architecture.md ## Non-goals`). *Zero-dependency:* spawning uses the Node stdlib; nothing is added to runtime `dependencies`. Tests inject a **stub** `AgentRunner` (records the spec, returns a chosen exit code) so executor/guard tests never spawn a real process.

### D4 ‚Äî One guard module for the whole `readOnlyGuards` union + `requiredOutputs`, driven by catalog data
`runner/guards.ts` provides `snapshot(guards, resolvedProtectedFiles) ‚Üí GuardSnapshot` and `verify(snapshot, resolvedProtectedFiles) ‚Üí { diagnostics[] }` plus `assertRequiredOutputs(requiredOutputs, bindings) ‚Üí { diagnostics[] }`, driven by the typed `ReadOnlyGuard` union already on the catalog (`sequencing/step-catalog.ts`), replacing the hand-copied `mktemp`/`cp`/`cmp -s` blocks and 7.1's bespoke exists/absent handling (`02_responsibility_translation_map.md` row 11):
- **`fromContext`** (steps 6/7/8/8.x): the protected set is the read-only inputs the context result reports (see D5); snapshot each, verify byte-identical after the session, diagnostic on any modification ‚Äî parity with the per-file `cmp -s` loop.
- **`mustExistUnchanged`** (steps 5/5.1 on `feature_br_summary.md`): named files must exist before and be byte-identical after; a missing-before or modified-after file is a diagnostic.
- **`preserveExistence`** (step 7.1 on `external_sources.yaml` + `init_progress_definition.yaml`): unchanged if present, absent must stay absent, never created/deleted/replaced ‚Äî the three-way check the shell hand-wrote for 7.1, now the only step with this mode.
- **`requiredOutputs`**: after a successful session, assert each declared output file exists; **an empty list is legal** and asserts nothing (7.1 mutates surface maps in place). *Why:* one implementation tested once, applied uniformly ‚Äî 7.1 becomes catalog data (already present in Slice 1's `STEP_CATALOG`), not an executor branch.

### D4a ‚Äî Correct catalog guard metadata so execution can stay data-driven
To let the executor and guard module trust catalog data without step-id branches, Slice 2 corrects the `sequencing/step-catalog.ts` guard metadata to match the shell's actual behavior. The default `fromContext` assignment is removed from generic session construction; guards are then attached only where shell parity requires them: steps 5/5.1 use `mustExistUnchanged` for `feature_br_summary.md`, steps 6/7/8/8.x use `fromContext`, and step 7.1 keeps `preserveExistence` but with runtime-correct relative paths (`.setup/external_sources.yaml`, `../init_progress_definition.yaml`). *Why:* the executor is specified to branch only on action kind, so incorrect catalog metadata would force hidden step-specific code paths or silently wrong guard coverage.

### D5 ‚Äî Extend `context/` to expose `readOnlyInputs` as a typed field ‚Äî no stdout parsing for the guard
The `fromContext` guard needs the read-only input paths the context builder computed. Today those paths exist **only** as `- read_only_input: <path>` lines inside `ContextResult.text`; the shell extracted them with `sed`. The architecture's reuse claim (`02_responsibility_translation_map.md` row 10: "reads `readOnlyInputs` from the typed result. Zero output parsing") is not yet met by the type. This slice therefore **additively** extends `ContextResult` with an optional structured `readOnlyInputs: string[]` and populates it in the builders that carry read-only inputs (the `fromContext` steps: contract-delta, surface-map, technical-requirements, implementation-slices, prerequisite-gaps, implementation-plan, plan-semantic-review, surface-map-enrich). The existing `.text` and its `- read_only_input:` lines are left byte-unchanged (the standalone `context` CLI verb and any consumer stay intact); the field is the same list the text is rendered from. The executor's `fromContext` guard reads `result.readOnlyInputs`, never the text. *Why:* honoring the "errors are values, zero output parsing" architecture principle at the seam that needs it, with the smallest additive change to an "Existing, reuse in-process" module (map row 10). *Alternative rejected:* porting the `sed` line-scrape into TS would re-introduce exactly the stdout-scraping seam the migration exists to remove.

### D6 ‚Äî `executeStep(stepDef, bindings)` iterates actions with no per-step branches; D7 ordering preserved
`executeStep(stepDef, bindings, deps) ‚Üí StepResult` iterates `stepDef.actions` in order (`02_responsibility_translation_map.md` row 5 executor; `03_target_architecture.md ## Runner`). `deps` injects the ports/functions (`AgentRunner`, the config loader, and the in-process `sync`/`context`/`readiness` functions) so the executor is unit-testable with stubs. Per action:
- **session** ‚Üí evaluate `runIf` (a named predicate from the closed set, e.g. `hasReadyClassRepo`, against workspace state; false ‚áí skip with a notice, not a failure) ‚Üí if `requiresSync`, run the in-process sync function (D7: repo-mutating, in the operator process, **before** the session) ‚Üí run the in-process context builder (read-only), obtaining `readOnlyInputs` (D5) ‚Üí load model config for `modelPhase` (D1) ‚Üí snapshot guards (D4) ‚Üí `buildSessionPrompt` (D2) ‚Üí `AgentRunner.run()` (D3) ‚Üí verify guards + assert required outputs (D4). A non-zero agent exit, a sync/context failure, or a guard/output violation stops the step with a typed `StepResult` carrying diagnostics; guard violations are checked **regardless** of the agent exit code, matching the shell (which verifies guards before returning the model rc).
- **deterministic** (`check` | `write`) ‚Üí call the named in-process coordinator function (e.g. `br-clarification` readiness for 4.2's `check`); a non-`check`/`write` name or an unregistered function is a diagnostic. The scaffold-feature `write` primitive itself lands in Slice 3 ‚Äî this slice wires the dispatch and covers it with a stub, since step 3's real primitive is a Slice 3 deliverable.
*Bindings scope this slice:* feature path and a single class (step 7). The **class-list** binding (reconciliation) is Slice 4; the executor contract is designed to extend to it (a list-valued binding) without a fork, per `03_target_architecture.md ## Decisions` decision 3 ("extend the executor contract ‚Ä¶ never fork a one-off launcher"). *Gate ownership unchanged:* the executor never runs `overmind gate`; the model owns the gate loop (`02_responsibility_translation_map.md` guiding rule).

### D7 ‚Äî `interaction/` port + TTY adapter now; consumed in Slice 3
`InteractionPort` with typed `confirm(message) ‚Üí boolean`, `select(message, options) ‚Üí choice`, `input(message) ‚Üí string` request shapes, plus a TTY adapter over `readline`/stdin preserving today's prompt wording and semantics (`02_responsibility_translation_map.md` row 14). It ships in this slice ‚Äî even though no Slice 2 module consumes it ‚Äî so the runner result and interaction shapes co-evolve rather than being retrofitted when the loop lands (`04_migration_plan.md ## Slice 2`: "needed by Slice 3; specified here so the runner result/interaction shapes co-evolve"). *Why:* the executor's typed `StepResult` and the interaction request shapes are the two halves of the operator seam; defining both now keeps Slice 3 additive. *Non-goal:* no auto-advance profiles (configuration for a later slice), no decision-point wiring (Slice 3 orchestrator).

### D8 ‚Äî No shell edits, no deletions this slice
Unlike Slice 1's transitional rewire, Slice 2 touches no shell: the e2e keeps `load_model_config`, its heredocs, its launchers, and its `.setup/models.md` reads until Slice 3's `overmind run` replaces the whole loop and deletes them (`04_migration_plan.md ## Slice 2`: "Deletes nothing yet"). *Why:* the clean-break rule deletes migrated shell in the slice that proves parity; Slice 2 ships reusable modules with no consumer wired into the live path, so nothing is yet proven replaceable. The parity is proven by tests (config, prompt, guards, executor over a stub agent), not by cutting over the operator flow.

## Risks / Trade-offs

- **Prompt parity drift** ‚Üí the 13 heredocs differ subtly (some emit `Use the ‚Ä¶ skill`, others `Load and follow the ‚Ä¶ skill`; some pass `--class`, some a sync command, 4.2 a readiness command). The parity test enumerates all 13 phases and asserts skill name + binding lines + exact command lines + no final-response lines; a divergence fails the test, not the operator's run.
- **`fromContext` protected-set mismatch** ‚Üí the guard trusts the context result's `readOnlyInputs` (D5). A test asserts, for a representative `fromContext` step, that `readOnlyInputs` equals the set the `- read_only_input:` text lines render (so the additive field and the text never diverge), and that a modified protected file is caught.
- **7.1 boundary regressions** ‚Üí the `preserveExistence` + empty-`requiredOutputs` shape is the one novel guard combination. Dedicated tests cover all three sub-cases (present-unchanged passes, present-modified fails, absent-created fails, absent-stays-absent passes) plus that an empty `requiredOutputs` asserts nothing ‚Äî the shell's bespoke 7.1 handling, now uniform.
- **Executor stubbing hides a real spawn bug** ‚Üí the `CodexAgentRunner` adapter is thin (spawn + exit code) and separately covered by an adapter-level test that the constructed `codex -m <model> <args...> <prompt>` argv and `cwd` match the shell's `cmd=(codex -m ‚Ä¶ ) ; cd runtime_root`; executor logic tests use the stub.
- **Context extension breaks the standalone `context` verb** ‚Üí D5 is strictly additive (new optional field; `.text` byte-unchanged). A test asserts the existing context text output is unchanged for a touched builder, so the live `context` CLI consumer is unaffected.
- **Scope creep into the orchestrator** ‚Üí `interaction/` ships unconsumed and `executeStep` is not called by any loop; any temptation to add feature selection, `PhaseOutcome`, or checkpoint commits is out of scope (Slice 3).

## Migration Plan

1. **Responsibility inventory** for the touched rows of `02_responsibility_translation_map.md` (rows 5 [executor], 6, 7, 8, 10 [reuse, extended], 11, 14 [partial]) ‚Äî confirm every behavior has a named TS owner in this slice; any unowned behavior blocks the slice.
2. **Audit the 13 heredocs + launchers** in `project_add_feature_e2e.sh`: record per phase the skill name, the `Use`/`Load` verb, the runtime-binding lines, the exact CLI command lines, whether a pre-session sync and/or context runs, the `readOnlyGuards` mode, and the `requiredOutputs` ‚Äî the parity fixture for D2/D4.
3. **Land `config/`:** `loadRunnerConfig` + validation + diagnostics; tests (well-formed, missing file, absent phase, short row, non-`codex` command).
4. **Extend `context/` (D5):** additive `readOnlyInputs: string[]` on `ContextResult`, populated in the `fromContext` builders; test that text is unchanged and the field matches the rendered lines.
5. **Land `runner/`:** prompt builder (D2) ‚Üí guards (D4) ‚Üí `AgentRunner` port + `CodexAgentRunner` (D3) ‚Üí generic `executeStep` (D6); each with tests, the executor over a stub agent.
6. **Land `interaction/` (D7):** port types + TTY adapter, with a light adapter test.
7. **Verify:** `npm run verify` green (TS workspaces + surviving `tests/ai_scripts/*.sh`, unchanged this slice); confirm runtime `dependencies` still empty.

**Rollback:** revert the change; nothing in the live operator path changed (no shell edits, no deletions, no new wired CLI verb), so rollback is a pure code removal with no state or format to unwind.
