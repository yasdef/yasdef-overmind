## Why

CRP-129 landed the TypeScript foundation (`asdlc-coordinator` + the `overmind capture/context/gate` CLI) and migrated `task-to-br`; CRP-130 corrected runner-skill installation; CRP-131 migrated the repo-scan half of step 4.1 and added the `overmind sync` verb plus the shared `repo/` ready-path + sync modules. The next step in the agreed migration (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md`) is **step 4.2 — "BR clarification + EARS readiness"** — still implemented as two bash scripts: `feature_user_br_clarification.sh` (model-loop orchestrator) + `user_br_clarification_rule.md` (rule) + `check_user_br_clarification_quality.sh` (gate), and the deterministic `feature_br_check_ears_readiness.sh` (no rule, no model — pure mechanics). This change migrates that step to the agent-skill + TypeScript-core pattern proven by CRP-129/130/131, and folds in the matching installer/setup/e2e wiring so no follow-up correction is needed.

Step 4.2 has **two distinct halves** with different ownership:
- **BR clarification** is a *model-owned loop*: ask business-only follow-up questions for the unresolved items tracked in `missing_br_data.md`, write answers into `feature_br_summary.md`, flip each `rised_item_N` to `rised=true`, and rerun the gate until every tracked item is resolved (skip cleanly when none are unresolved). → an agent **skill** (`overmind-br-clarification`) plus a structural **gate** in the core.
- **EARS readiness** is a *deterministic transition* (never a model phase): verify the `task-to-br` gate passes, verify the `repo-br-scan` gate passes when ready class repos exist, then flip `ready_to_ears: false → true` in `## 1. Document Meta`. → a deterministic **`overmind readiness` CLI verb** in the core (mirroring how CRP-131 added the deterministic `sync` verb), invoked by the orchestrator after the clarification skill completes — not by the model.

## What Changes

- Add the **`br-clarification` step** to the shared `asdlc-coordinator` core:
  - `validate/br-clarification.ts` — structural gate with behavior parity to `check_user_br_clarification_quality.sh`: run the existing `task-to-br` validator as the base check and surface any base failure verbatim; if it passes, parse `missing_br_data.md` `## 3. Unresolved Items Ledger (Rised)` and fail (exit `1`, actionable `missing:` line) when any tracked `- rised_item_N:` entry is explicitly unresolved (`rised=false`); exit `2` on runtime failure. Invalid or missing `rised` markers remain `task-to-br` structural failures for the skill to repair by adding `rised=false`. Registered in the `gate` dispatch registry. (Naturally covers the bash "skip when no non-rised items" case: an empty/all-rised ledger passes immediately.)
  - `context/br-clarification.ts` — deterministic context assembly with parity to `feature_user_br_clarification.sh`'s prompt context: resolve the feature path, target `feature_br_summary.md`, and `missing_br_data.md`; emit one assembled context block (workspace root, target artifacts, allowed-write list, skill-relative asset references — the rule is inlined in `SKILL.md`, not referenced as a separate file, exact gate command). Registered in the `context` dispatch registry.
  - Add the **`readiness` CLI verb** with a `readiness/br-clarification.ts` (a.k.a. EARS-readiness) handler porting `feature_br_check_ears_readiness.sh`. It evaluates its preconditions by calling the shared validator **functions** in-process (`validateBrClarification`, `validateRepoBrScan`) — **not** by shelling out to the `overmind gate` CLI verb, which stays exclusively model-invoked (rule 4); the function reuse is the deterministic-validation path sanctioned by rule 2. It evaluates the `br-clarification` validator (must pass) as the business-context precondition — a documented superset of the old bare `task-to-br` check (it adds the unresolved-ledger check), so that any unanswered or `skip for now`-deferred clarification item blocks `ready_to_ears` from flipping, preserving the old invariant that the clarification phase could not complete with unresolved items; resolves the project's `init_progress_definition.yaml` and, when any `meta_info.class_repo_paths` entry is `ready` (reusing the CRP-131 `repo/collect-ready-paths` module), evaluates the `repo-br-scan` validator (must pass) — otherwise prints the skip notice; then flips `ready_to_ears` `false → true` in `## 1. Document Meta` of `feature_br_summary.md`, validating the precondition that it is currently `false`. Replace the `readinessStub` placeholder. Add a small BR-summary meta parse/update helper under `parse/`. Registered in a new `readinessRegistry` and wired into the CLI dispatch (`capture|context|gate|sync|readiness`).
- Add the **`overmind-br-clarification` skill** (`packages/installer/_data/skills/overmind-br-clarification/SKILL.md` with `user_br_clarification_rule.md` inlined; `assets/` BR-summary template + golden example). The model-owned loop: run `overmind context br-clarification <feature-path>` → show the full list of unresolved business questions + a brief process note, then ask them **one at a time** (each question waits for the user's reply before the next), where the user may answer or reply `skip for now` to defer a question (a skipped item stays `rised=false`) → write answers only to `feature_br_summary.md`, record pointer-only `- answers:` entries and `rised=true` transitions in `missing_br_data.md`, append qualifying links to `## 16. Linked Artifacts` → run `overmind gate br-clarification <feature-path>` after every round → finish only on `0` (every item answered); on `1`, keep working the loop (re-offer the deferred `skip for now` questions, repair) **without** declaring completion or advancing to the next phase; stop-and-ask on `2`. A `skip for now` defers a question within the loop (answer the rest first, come back to it); it does not finish the phase — the phase keeps running until all items are `rised=true`, and if the session ends with anything unresolved the readiness/next phase does not run (matching the old hard-fail-on-unresolved behavior). The literal final response line (`User BR clarification phase is finished. ... press Ctrl-C so orchestrator can start the next phase`) lives only in `SKILL.md`.

The launcher stays a thin prompt; model-facing semantic instructions live only in `SKILL.md`.
