## Why

CRP-129 landed the TypeScript foundation (`asdlc-coordinator` + the `overmind capture/context/gate` CLI) and migrated `task-to-br`; CRP-130 corrected runner-skill installation; CRP-131 migrated `repo-br-scan` and added the `overmind sync` verb; CRP-132 migrated step 4.2 (BR clarification + EARS readiness) with the `overmind readiness br-clarification` verb; CRP-133 migrated step 5 (BR → EARS) as the `overmind-requirements-ears` skill + `requirements-ears` gate/context, and explicitly left step 5.1 untouched as a separate un-migrated step. The next row in the agreed migration (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md` Steps→Skills table) is **step 5.1 — "EARS review (opt)"** → `overmind-ears-review` — still implemented as bash: `feature_requirements_ears_review.sh` (model orchestrator) + `requirements_ears_review_rule.md` (rule) + `check_requirements_ears_review_quality.sh` (gate helper). This change migrates that optional step to the agent-skill + TypeScript-core pattern proven by CRP-129/131/132/133, and folds in the matching installer/setup/e2e wiring so no follow-up correction is needed.

Step 5.1 is an **optional, user-interactive** model step: it compares the EARS requirements (`requirements_ears.md`) against the validated `feature_br_summary.md` for **material business gaps only**, surfaces findings one at a time (highest severity first), shows the operator a fixed 3-line finding/recommendation/decision prompt, applies accepted changes, and maintains a durable findings ledger in `requirements_ears_review.md`. It then runs a structural quality gate over the **review ledger** and repairs until it passes (or stops with an explicit "cannot be completed with current BR/EARS input" line when completion is infeasible). Unlike step 5, this step **writes two artifacts** (`requirements_ears.md` and `requirements_ears_review.md`), treats `feature_br_summary.md` as read-only, and has a **user-question loop** (like step 4.2) rather than a deterministic transition. There is **no capture** (the BR summary and EARS file are upstream artifacts) and **no readiness verb** (this step has no state flip of its own).

## What Changes

- Add the **`ears-review` step** to the shared `asdlc-coordinator` core:
  - `validate/ears-review.ts` — structural gate with behavior parity to `check_requirements_ears_review_quality.sh`, validating `requirements_ears_review.md`: fail (exit `1`) on an empty target or any remaining `[UNFILLED]` placeholder; require sections `## 1. Document Meta`, `## 2. Review Guidance`, `## 3. Findings Ledger`; require meta keys `feature_id`, `feature_title`, `source_feature_br_summary`, `source_requirements_ears`, `review_status`, `last_updated` (present and filled); `review_status` ∈ {`in_progress`, `complete`}; parse `### Finding <N> — <Title>` blocks and require each to contain the 10 fields (`severity`, `state`, `source_br_summary_reference`, `related_requirement_targets`, `gap_summary`, `recommendation`, `suggested_ears_change`, `user_prompt`, `user_response`, `resolution_notes`); `severity` ∈ {`High`, `Medium`, `Low`}; `state` ∈ {`escalated`, `added to ears`, `rejected`, `postponed`}; enforce the cross-field rules — when no Finding blocks exist the ledger must declare `- no_findings: true` and `review_status: complete`; `no_findings: true` is rejected when Finding blocks are present; `review_status: complete` is rejected when any finding is `escalated`; `review_status: in_progress` with findings but none `escalated` fails; exit `0` on pass, `2` on runtime failure. Registered in the `gate` dispatch registry.
  - `context/ears-review.ts` — deterministic context assembly with parity to `feature_requirements_ears_review.sh`'s prompt context: resolve the feature path, the read-only `feature_br_summary.md` source (exit `2` if absent), the `requirements_ears.md` source/target (exit `2` if absent), and the `requirements_ears_review.md` ledger target; emit one context block with the workspace root, feature artifact root, the read-only BR source, the two allowed-write targets (`requirements_ears.md` and `requirements_ears_review.md`), skill-relative asset references (review template + golden example), and the exact `ears-review` gate command. Registered in the `context` dispatch registry.
- Add the **`overmind-ears-review` skill** (`packages/installer/_data/skills/overmind-ears-review/SKILL.md` with `requirements_ears_review_rule.md` inlined; `assets/` review template + golden example). The model-owned loop: run `overmind context ears-review <feature-path>` → compare `requirements_ears.md` against `feature_br_summary.md` for material business findings only → ask the operator one finding at a time (highest severity first) using the exact 3-line interaction format → apply accepted edits to `requirements_ears.md` and keep `requirements_ears_review.md` synchronized with the decision history (each finding in exactly one of `escalated`/`added to ears`/`rejected`/`postponed`) → write only `requirements_ears.md` and `requirements_ears_review.md`, never modifying `feature_br_summary.md` → run `overmind gate ears-review <feature-path>` after every write, repair on `1`, stop-and-ask on `2`, finish on `0` when no finding remains `escalated`. The two literal final-response lines live only in `SKILL.md`: the success line `requirements_ears extra review phase is finished. Nothing else to do now; press Ctrl-C so orchestrator can start the next phase` and the infeasibility line `based on provided reasons, requirements_ears extra review cannot be completed with current BR/EARS input. Please provide instructions what to do, or adjust artifacts and rerun this phase`.
- **Fold in installer + setup + e2e wiring (so no follow-up is needed):**
  - Extend `packages/installer/src/init.ts`'s `PACKAGED_SKILLS` set to include `overmind-ears-review` (installed to `.codex/skills/` and `.claude/skills/`), `.overmind/overmind.js` staying the single shared CLI.
  - Extend skill-staging in `overmind/scripts/project_mgmt/project_setup_first_init_machine.sh` to stage `overmind-ears-review`, and **remove** staging of the migrated `feature_requirements_ears_review.sh` command, `requirements_ears_review_rule.md` rule, and `check_requirements_ears_review_quality.sh` helper (update mode cleans up the stale staged copies).
  - Rewire `overmind/scripts/project_mgmt/project_add_feature_e2e.sh` phase 5.1 to drive the new skill via a Codex session (`run_ears_review_skill` / `build_ears_review_prompt`, mirroring the phase 4.2 / phase 5 launchers) instead of running `feature_requirements_ears_review.sh`. The launcher stays a thin prompt; model-facing semantic instructions and literal final lines live only in `SKILL.md`. The launcher retains the old script's deterministic read-only-BR protection (`ensure_feature_br_summary_unchanged`): it snapshots `feature_br_summary.md` before the session and `cmp`-asserts it unchanged after, failing the phase if the model mutated it.
- **Retrofit the same deterministic guard onto the already-shipped step 5 phase 5 launcher (fixes a CRP-133 gap):** CRP-133 migrated step 5 (BR→EARS) but dropped `feature_br_to_ears.sh`'s `ensure_feature_br_summary_unchanged` protection, leaving `run_requirements_ears_skill` with only advisory allowed-write/SKILL.md text. Add the same snapshot-before / `cmp`-assert-after read-only-BR guard to `run_requirements_ears_skill` in `project_add_feature_e2e.sh` so a model that mutates `feature_br_summary.md` during the BR→EARS phase fails the phase loudly instead of silently corrupting the approved BR for all downstream steps.
- Port the gate/context scenarios from `tests/ai_scripts/check_requirements_ears_review_quality_tests.sh` and `tests/ai_scripts/init_feature_requirements_ears_review_tests.sh` to the TS test runner, and update `project_setup_asdlc_tests.sh` / `project_add_feature_e2e_tests.sh` to the migrated wiring.
- **BREAKING / clean break:** remove `overmind/scripts/feature_requirements_ears_review.sh`, `overmind/rules/requirements_ears_review_rule.md`, `overmind/scripts/helper/check_requirements_ears_review_quality.sh`, `tests/ai_scripts/init_feature_requirements_ears_review_tests.sh`, and `tests/ai_scripts/check_requirements_ears_review_quality_tests.sh`. No dual bash+TS path. The migrated step 5 (`requirements-ears`) and all later un-migrated steps (6 contract-delta onward) stay untouched.
