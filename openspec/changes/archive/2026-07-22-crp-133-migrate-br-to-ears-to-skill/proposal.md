## Why

CRP-129 landed the TypeScript foundation (`asdlc-coordinator` + the `overmind capture/context/gate` CLI) and migrated `task-to-br`; CRP-130 corrected runner-skill installation; CRP-131 migrated `repo-br-scan` and added the `overmind sync` verb; CRP-132 migrated step 4.2 (BR clarification + EARS readiness), folding in the `overmind readiness br-clarification` verb that flips `ready_to_ears: false → true`. The next step in the agreed migration (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md` Steps→Skills table) is **step 5 — "BR → EARS"** — still implemented as bash: `feature_br_to_ears.sh` (model orchestrator) + `br_to_ears.md` (rule) + `check_requirements_ears_quality.sh` (gate helper). This change migrates that step to the agent-skill + TypeScript-core pattern proven by CRP-129/131/132, and folds in the matching installer/setup/e2e wiring so no follow-up correction is needed.

Step 5 is a **single model-owned step** (unlike 4.2's two halves): it reads the readiness-approved `feature_br_summary.md` and writes deterministic, testable, business-facing EARS requirements to `requirements_ears.md`, then runs a structural quality gate and repairs until it passes (or stops with an explicit "gate cannot pass with current BR input" line when conversion is infeasible). The `ready_to_ears: true` precondition that the old script enforced (`ensure_ready_to_ears`) is now produced upstream by CRP-132's `readiness br-clarification` flip; this step only verifies it before running. There is **no capture** — the BR summary is an upstream artifact, not user-captured input.

## What Changes

- Add the **`requirements-ears` step** to the shared `asdlc-coordinator` core:
  - `validate/requirements-ears.ts` — structural gate with behavior parity to `check_requirements_ears_quality.sh`: fail (exit `1`) on an empty target; parse `### Requirement <N> — <Title>` / `### NFR <N> — <Title>` blocks and require each block to contain `**User Story:**`, `**Acceptance Criteria (EARS):**` (with at least one bullet and at least one valid EARS-pattern bullet), and `**Verification:**`; enforce sequential 1-based numbering with no duplicates for `Requirement` and `NFR` independently; reject EARS bullets that do not match an allowed pattern (`WHEN …, THE … SHALL …`; `IF …, THEN THE … SHALL …`; `WHILE …, THE … SHALL …`; `WHERE …, THE … SHALL …`; the `WHEN … AND WHILE …` combined form; or a bare `THE … SHALL …`); fail when no `Requirement`/`NFR` blocks are found; exit `0` on pass, `2` on runtime failure. Registered in the `gate` dispatch registry.
  - `context/requirements-ears.ts` — deterministic context assembly with parity to `feature_br_to_ears.sh`'s prompt context: resolve the feature path, the read-only `feature_br_summary.md` source (exit `2` if absent), and the `requirements_ears.md` target; verify the `ready_to_ears: true` precondition in `## 1. Document Meta` of `feature_br_summary.md` and exit `2` with a "run `readiness br-clarification` first" message when it is missing or not `true` (porting `ensure_ready_to_ears`); emit one context block with the workspace root, feature artifact root, allowed-write list (`requirements_ears.md` only), skill-relative asset references (template + golden example), and the exact `requirements-ears` gate command. Registered in the `context` dispatch registry.
  - Reuse the CRP-132 `parse/` BR-summary meta reader for the `ready_to_ears` precondition (no new meta parser; read-only — this step does not flip meta).
- Add the **`overmind-requirements-ears` skill** (`packages/installer/_data/skills/overmind-requirements-ears/SKILL.md` with `br_to_ears.md` inlined; `assets/` EARS template + golden example). The model-owned loop: run `overmind context requirements-ears <feature-path>` → convert the BR summary into atomic EARS `Requirement`/`NFR` blocks using only facts present in the BR (marking necessary inferences `[Inference]`, recording `Unresolved gap:` notes, never starting a new question loop), propagating `## 16. Linked Artifacts` per the link rules → write only `requirements_ears.md`, never modifying `feature_br_summary.md` → run `overmind gate requirements-ears <feature-path>` after every write, repair on `1`, stop-and-ask on `2`, finish on `0`. The two literal final-response lines live only in `SKILL.md`: the success line `BR->requirement-EARS phase is finished. Nothing else to do now; press Ctrl-C so orchestrator can start the next phase` and the infeasible line `based on provided reasons, EARS gate cannot pass with current BR input. Please provide instructions what to do, or adjust requirements and rerun this phase`.
- **Fold in installer + setup + e2e wiring (so no follow-up is needed):**
  - Extend `packages/installer/src/init.ts`'s `PACKAGED_SKILLS` set to include `overmind-requirements-ears` (installed to `.codex/skills/` and `.claude/skills/`), `.overmind/overmind.js` staying the single shared CLI.
  - Extend skill-staging in `overmind/scripts/project_mgmt/project_setup_first_init_machine.sh` to stage `overmind-requirements-ears`, and **remove** staging of the migrated `feature_br_to_ears.sh` command, `br_to_ears.md` rule, and `check_requirements_ears_quality.sh` helper (update mode cleans up the stale staged copies).
  - Rewire `overmind/scripts/project_mgmt/project_add_feature_e2e.sh` phase 5 to drive the new skill via a Codex session (`run_requirements_ears_skill` / `build_requirements_ears_prompt`, mirroring the phase 4.2 launcher) instead of running `feature_br_to_ears.sh`. The launcher stays a thin prompt; model-facing semantic instructions and literal final lines live only in `SKILL.md`.
- Port the gate/context scenarios from `tests/ai_scripts/check_requirements_ears_quality_tests.sh` and `tests/ai_scripts/init_br_to_ears_tests.sh` to the TS test runner, and update `project_setup_asdlc_tests.sh` / `project_add_feature_e2e_tests.sh` to the migrated wiring.
- **BREAKING / clean break:** remove `overmind/scripts/feature_br_to_ears.sh`, `overmind/rules/br_to_ears.md`, `overmind/scripts/helper/check_requirements_ears_quality.sh`, `tests/ai_scripts/init_br_to_ears_tests.sh`, and `tests/ai_scripts/check_requirements_ears_quality_tests.sh`. No dual bash+TS path. Step 5.1 (`feature_requirements_ears_review.sh` and its `check_requirements_ears_review_quality.sh`) is a **separate** un-migrated step and stays untouched.
