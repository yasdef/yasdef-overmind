## Why

CRP-129 landed the TypeScript foundation (`asdlc-coordinator` + the `overmind capture/context/gate` CLI) and migrated `task-to-br`; CRP-130 corrected runner-skill installation; CRP-131 migrated `repo-br-scan` and added the shared repo machinery (`collectReadyRepoPaths`, `checkRepoBranchState`, `syncRepoToDefaultBranch`) plus the `overmind sync` verb; CRP-132 migrated step 4.2 (BR clarification + EARS readiness); CRP-133 migrated step 5 (BR → EARS); CRP-134 migrated step 5.1 (EARS review) as `overmind-ears-review`. The next row in the agreed migration (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md` Steps→Skills table) is **step 6 — "Contract delta"** → `overmind-contract-delta` — still implemented as bash: `feature_contract_delta.sh` (model orchestrator) + `feature_contract_delta_rule.md` (rule) + `check_feature_contract_delta_quality.sh` (gate helper). This change migrates that step to the agent-skill + TypeScript-core pattern proven by CRP-129/131/132/133/134, and folds in the matching installer/setup/e2e wiring so no follow-up correction is needed.

Step 6 converts feature-level EARS requirements into a contract delta artifact (`feature_contract_delta.md`) against the stable project baseline (`common_contract_definition.md`). Unlike step 5.1, it is a **deterministic single-target model step** (like step 5: no user-question loop), but it has the richest **deterministic context** of any step migrated so far: it scans the project's **ready class repositories** (`meta_info.class_repo_paths` with `state: ready`), syncs each to its default branch before the session, collects **pending sibling contract deltas** (committed sibling features' `feature_contract_delta.md`) as additional read-only inputs, and computes a **cross-class peer trigger** that decides whether the optional `## 5. Cross-Class Transport/Contract Approach Mirror` section applies. It writes a single artifact (`feature_contract_delta.md`) and treats four input classes as read-only: `feature_br_summary.md`, `requirements_ears.md`, `common_contract_definition.md`, and every pending sibling `feature_contract_delta.md`. There is **no capture** (BR/EARS/common-contract are upstream artifacts) and **no readiness verb** (this step has no state flip of its own).

## What Changes

- Add the **`contract-delta` step** to the shared `asdlc-coordinator` core:
  - `validate/contract-delta.ts` — structural gate with behavior parity to `check_feature_contract_delta_quality.sh`, validating `feature_contract_delta.md`: fail (exit `1`) on an empty/whitespace target or any remaining `[UNFILLED]` placeholder; require sections `## 1. Document Meta`, `## 2. Delta Summary`, `## 3. Contract Delta Items`, `## 4. Track Handoff Signals`; require meta keys `feature_id`, `feature_title`, `project_type_code`, `source_requirements_ears`, `source_common_contract_definition`, `delta_needed`, `last_updated` (present and filled); `delta_needed` ∈ {`true`, `false`}; when `delta_needed: true` require ≥1 `### Delta N:` block, reject `- no_contract_delta_required: true`, and require each Delta block to carry the five fields `delta_kind`, `related_baseline_contract`, `change_scope`, `compatibility_impact`, `verification_expectation` (present and filled); when `delta_needed: false` require the exact line `- no_contract_delta_required: true` and reject any present Delta block; require handoff keys `backend_handoff` and `frontend_mobile_handoff` (present and filled); exit `0` on pass, `2` on runtime failure. The `## 5. Cross-Class Transport/Contract Approach Mirror` section is **not gate-enforced** (parity with the rule's "no quality-gate enforcement for §5"). Registered in the `gate` dispatch registry.
  - `context/contract-delta.ts` — deterministic context assembly with parity to `feature_contract_delta.sh`'s `build_prompt`: resolve the feature path and its `projects/<project-id>` root (exit `2` if not under `projects/<id>/<feature>`); resolve the read-only `feature_br_summary.md`, `requirements_ears.md`, and project-level `common_contract_definition.md` (exit `2` if any absent); resolve `init_progress_definition.yaml`; collect **ready** class repo paths via the shared `collectReadyRepoPaths` and run `checkRepoBranchState` on each (block/exit `2` verbatim on wrong-branch/dirty per the D7 read-only contract); collect **pending sibling contract deltas** (each committed sibling feature's `feature_contract_delta.md`, when present) as read-only inputs; compute the **cross_class_peer_trigger** (`active`/`inactive`) value; emit one context block with the workspace root, project root, feature root, one stable `- read_only_input: <workspace-relative-path>` manifest entry per resolved read-only input, the ready-repo list, the pending sibling-delta list, the `cross_class_peer_trigger` value, the single allowed-write target (`feature_contract_delta.md`), skill-relative asset references (template + golden example), and the exact `contract-delta` gate command. Registered in the `context` dispatch registry.
  - `sync/contract-delta.ts` — `overmind sync contract-delta <feature-path>` syncs every **ready** class repo to its default branch before the model session (reusing `collectReadyRepoPaths` + `syncRepoToDefaultBranch`, mirroring `sync/repo-br-scan.ts`), keeping the D7 boundary that writes (`git pull --rebase`) happen in the shell-invoked `sync` verb, not in `context`. Registered in the `sync` dispatch registry.
  - Two **reusable** shared modules ported from the retained shell helpers: `repo/list-committed-sibling-features.ts` (committed siblings = sibling feature folders that contain `implementation_plan.md`, parity with `common_libs/list_committed_sibling_features.sh`) and `repo/cross-class-peer-trigger.ts` (active when `project_type_code == A` and backend is present and either another class exists or there is more than one backend, parity with `helper/check_cross_class_peer_trigger.sh`).
- Add the **`overmind-contract-delta` skill** (`packages/installer/_data/skills/overmind-contract-delta/SKILL.md` with `feature_contract_delta_rule.md` inlined; `assets/` template + golden example). The model-owned loop: run `overmind context contract-delta <feature-path>` → compare `requirements_ears.md` against the `common_contract_definition.md` baseline for feature-level contract additions/changes only → draft `feature_contract_delta.md` from the template (set `delta_needed` true/false, one `### Delta N:` block per independent delta, or `- no_contract_delta_required: true`) → honor the cross_class_peer_trigger value from context for the optional `## 5.` section (omit the section entirely when `inactive`) → report (not resolve) overlaps against pending sibling contract deltas → write only `feature_contract_delta.md`, never modifying the read-only inputs → run `overmind gate contract-delta <feature-path>` after every write, repair on `1`, stop-and-ask on `2`, finish on `0`. The two literal final-response lines live only in `SKILL.md`: the success line `Feature contract delta phase is finished. Nothing else to do now; press Ctrl-C so orchestrator can start the next phase` and the infeasibility line `feature contract delta gate cannot pass with current EARS/common-contract inputs. Please provide instructions what to do, or adjust requirements and rerun this phase`.
- **Fold in installer + setup + e2e wiring (so no follow-up is needed):**
  - Extend `packages/installer/src/init.ts`'s `PACKAGED_SKILLS` set to include `overmind-contract-delta` (installed to `.codex/skills/` and `.claude/skills/`), `.overmind/overmind.js` staying the single shared CLI.
  - Extend skill-staging in `overmind/scripts/project_mgmt/project_setup_first_init_machine.sh` to stage `overmind-contract-delta`, and **remove** staging of the migrated `feature_contract_delta.sh` command, `feature_contract_delta_rule.md` rule, and `check_feature_contract_delta_quality.sh` helper (update mode cleans up the stale staged copies). The contract-delta template and golden example are staged only inside the skill's `assets/`; their legacy flat `.templates/` and `.golden_examples/` copies are omitted on fresh setup and removed by the exact-manifest cleanup in update mode. The shared `check_cross_class_peer_trigger.sh` helper and `list_committed_sibling_features.sh` lib stay staged — they are still used by un-migrated steps (stack-blueprint, common-contract, prerequisite-gaps, surface-map).
  - Rewire `overmind/scripts/project_mgmt/project_add_feature_e2e.sh` phase 6 to drive the new skill via a Codex session (`run_contract_delta_skill` / `build_contract_delta_prompt`, mirroring the phase 4.1 repo-br-scan launcher that pre-syncs repos and the phase 5.1 launcher that snapshots read-only inputs) instead of running `feature_contract_delta.sh`. The launcher runs `overmind sync contract-delta` before the session, invokes `overmind context contract-delta` once for deterministic guard setup, and snapshots exactly the `read_only_input` manifest emitted by that context; it does not independently discover siblings through the retained shell lister. It stays a thin prompt (no literal final lines, no gate handling), retains the old script's deterministic read-only-input protection, and asserts each snapshotted file unchanged after the session. Context failure blocks before model launch. The `cmp` comparison runs on **every** model exit path — including when the model session exits non-zero — and is evaluated before the launcher returns the model's exit code, with the read-only-corruption error winning on a double-failure, so a model that both corrupts an input and fails cannot bypass the guard.
  - **Retrofit the same every-exit-path guard onto the shipped phase 5.1 ears-review launcher (fixes a CRP-134 gap):** `run_ears_review_skill` early-returns the model's non-zero exit code before reaching its `feature_br_summary.md` `cmp` check, so a model that corrupts the read-only BR and also fails bypasses the guard. Apply the identical "comparison runs on every exit path; read-only-corruption error wins on double-failure" fix to `run_ears_review_skill`, limited to the guard ordering (step 5.1's skill/gate/context/prompt/final lines unchanged), mirroring the CRP-134 D8 precedent that retrofitted the read-only-BR guard onto the phase 5 launcher.
- Port the gate/context scenarios from `tests/ai_scripts/check_feature_contract_delta_quality_tests.sh` and `tests/ai_scripts/init_feature_contract_delta_tests.sh` to the TS test runner, and update `project_setup_asdlc_tests.sh` / `project_setup_update_project_tests.sh` / `project_add_feature_e2e_tests.sh` to the migrated wiring.
- Refresh the repository-local `overmind-step-deployability` meta-skill so its reference deployable-step list no longer points to the deleted `overmind/scripts/feature_contract_delta.sh` path.
- **BREAKING / clean break:** remove `overmind/scripts/feature_contract_delta.sh`, `overmind/rules/feature_contract_delta_rule.md`, `overmind/scripts/helper/check_feature_contract_delta_quality.sh`, `tests/ai_scripts/init_feature_contract_delta_tests.sh`, and `tests/ai_scripts/check_feature_contract_delta_quality_tests.sh`. No dual bash+TS path. The shared cross-class/sibling helpers/libs and their tests, the migrated steps 4.1–5.1, and all later un-migrated steps (7 surface-map onward) stay untouched.
