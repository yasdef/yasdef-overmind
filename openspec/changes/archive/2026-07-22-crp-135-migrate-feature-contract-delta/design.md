## Context

Step 6 of the Overmind init/feature pipeline is "Contract delta". Today it is one bash script run by the e2e orchestrator (`project_add_feature_e2e.sh` phase 6 → `feature_contract_delta.sh`):

- `feature_contract_delta.sh` (+ `feature_contract_delta_rule.md` rule + `check_feature_contract_delta_quality.sh` helper) — a **deterministic single-target model step**: it resolves the feature path and its `projects/<project-id>` root; resolves the read-only `feature_br_summary.md`, `requirements_ears.md`, and project-level `common_contract_definition.md`; collects the project's **ready** class repos (`meta_info.class_repo_paths` with `state: ready`) and **syncs each to its default branch** (`common_libs/sync_repo_to_default_branch.sh`); collects **pending sibling contract deltas** by listing committed sibling features (`common_libs/list_committed_sibling_features.sh`, siblings that have `implementation_plan.md`) and adding each sibling's `feature_contract_delta.md` (when present) as a read-only input; loads the model config; builds a prompt binding the rule/template/golden-example + the quality-gate command + the ready-repo list + the pending-sibling-delta list + the cross-class peer trigger helper command; snapshots all read-only inputs, launches a Codex session, asserts the model produced `feature_contract_delta.md`, and `cmp`-asserts none of the read-only inputs changed (`ensure_readonly_inputs_unchanged`). The helper gate validates `feature_contract_delta.md` structure (sections, meta keys, the `delta_needed` true/false branch with per-Delta-block fields or the `- no_contract_delta_required: true` line, and the two handoff keys) with awk; it deliberately does **not** validate the optional `## 5.` section.

Two pieces of the old prompt are deterministic helpers, not model semantics: the **cross-class peer trigger** (`check_cross_class_peer_trigger.sh`, computing `active`/`inactive` from `init_progress_definition.yaml`) and the **committed-sibling lister** (`list_committed_sibling_features.sh`). Both shell helpers are **shared** — `check_cross_class_peer_trigger.sh` is also called by the un-migrated `init_project_stack_blueprints.sh` and `init_common_contract_definition.sh`, and `list_committed_sibling_features.sh` by the un-migrated `feature_prerequisite_gaps.sh` and `feature_repo_surface_and_exec_context.sh` — so they cannot be deleted by this change. Their logic is ported to reusable TS modules for the migrated step's context while the shell helpers stay in place for the un-migrated callers.

This change is the seventh in the md+sh → skill+TS migration, after CRP-129/130/131/132/133/134. The constraints from `design_docs/to_skills_migration/*` apply: clean break (no dual bash+TS path), the model owns the artifact loop, JS/TS owns deterministic mechanics, the gate is model-invoked with the `0/1/2` contract, repo writes (sync) live in the shell-invoked `sync` verb (D7), and skills install only to the supported `.codex`/`.claude` runner targets sharing the single `.overmind/overmind.js` CLI. The design overview Steps→Skills table names this skill `overmind-contract-delta`; gate name = skill name minus the `overmind-` prefix → `contract-delta`.

## Goals / Non-Goals

**Goals:**
- Migrate step 6 to the `overmind-contract-delta` skill backed by a `contract-delta` gate + context builder + `sync` verb in the core, with behavior parity to the old rule + helper + orchestrator.
- Preserve the single-target `feature_contract_delta.md` ownership, the `delta_needed` true/false branching, the §5 cross-class mirror behavior (driven by the computed trigger, omitted when inactive, gate-exempt), the read-only boundary over all four input classes, and the two terminal final-response lines.
- Reuse the CRP-131 repo machinery (`collectReadyRepoPaths`, `checkRepoBranchState`, `syncRepoToDefaultBranch`, the `sync` verb pattern) rather than re-implementing repo scanning, and port the two shared shell helpers to reusable TS without deleting them.
- Fold the installer/setup/e2e wiring into this same change so no CRP-130-style follow-up is needed.
- Port both bash test suites to TS-runner + shell-suite coverage, then delete the migrated step-6 bash, rule, helper, and old tests (clean break).

**Non-Goals:**
- The cross-step TS orchestrator / state machine (`overmind run`); the e2e shell launcher stays the transitional sequencer.
- Migrating `project_add_feature_e2e.sh` / `project_setup_first_init_machine.sh` themselves to TS.
- `.github`/`.agents` runner fan-out (still deferred).
- Step 7 (`feature_repo_surface_and_exec_context.sh`) and all later pipeline steps; migrating the un-migrated callers of the shared cross-class/sibling helpers.

## Decisions

### D1 — One skill + one gate + one context builder + one sync verb, no capture and no readiness verb
Step 6 has no user-captured input (BR/EARS/common-contract are upstream artifacts) and no state-flip of its own, and unlike step 5.1 it has no user-question loop. It needs the three standard model-facing primitives plus a repo-sync verb: the `overmind-contract-delta` skill (deterministic draft → gate loop), the model-invoked `gate contract-delta` (structural validation of `feature_contract_delta.md`), the deterministic `context contract-delta` (path bindings, read-only-input resolution, ready-repo branch-state checks, pending sibling-delta collection, computed trigger), and the shell-invoked `sync contract-delta` (ready-repo default-branch sync before the session). No `capture` and no `readiness` verb are added.

### D2 — The gate validates `feature_contract_delta.md` only, and is §5-exempt
The TS port of `check_feature_contract_delta_quality.sh` validates the four required sections, the seven meta keys, the `delta_needed` ∈ {true, false} branch (true ⇒ ≥1 Delta block each with five filled fields and no `no_contract_delta_required: true`; false ⇒ the exact `- no_contract_delta_required: true` line and no Delta blocks), and the two handoff keys — exactly as the awk does. It deliberately does **not** validate the optional `## 5. Cross-Class Transport/Contract Approach Mirror` section: the rule states there is "no resolution state machine, terminal-state field, required block, or quality-gate enforcement for §5". Keeping the gate §5-exempt preserves the rule's contract and keeps the gate idempotent across the model's repair loop.

_Alternative considered:_ also enforce §5 block shape. Rejected — it contradicts the rule's explicit §5 gate-exemption and would block legitimate `<to be defined during first feature implementation plan>` placeholders.

### D3 — `validate/contract-delta.ts` ports the awk validator one-for-one
The TS port reproduces the helper's checks exactly before any additions: empty/whitespace-target failure; `[UNFILLED]` placeholder failure; required sections (`## 1. Document Meta`, `## 2. Delta Summary`, `## 3. Contract Delta Items`, `## 4. Track Handoff Signals`); required filled meta keys (`feature_id`, `feature_title`, `project_type_code`, `source_requirements_ears`, `source_common_contract_definition`, `delta_needed`, `last_updated`); `delta_needed` lowercased ∈ {`true`, `false`}; the `delta_needed: true` branch (≥1 `### Delta N:` block parsed in section 3, `no_contract_delta_required` not `true`, each block carrying filled `delta_kind`/`related_baseline_contract`/`change_scope`/`compatibility_impact`/`verification_expectation`); the `delta_needed: false` branch (`no_contract_delta_required` == `true` required, zero Delta blocks); the two filled handoff keys (`backend_handoff`, `frontend_mobile_handoff`), all with the same key/value normalization the awk used (trim, quote-strip, `[UNFILLED]`/empty treated as unfilled). Exit codes map to the migration contract: structural failures → `1` with the same `quality gate failed: …` messages; runtime failures → `2`.

### D4 — The cross-class peer trigger is computed in `context`, not run by the model
The old rule told the model to "determine whether §5 applies by running the runtime-bound cross-class peer trigger helper". That is a deterministic computation over `init_progress_definition.yaml` (`project_type_code == A` and a backend is present and either another class exists or there is more than one backend), so per the migration ownership rules it belongs in TS, not in a model-run shell command. `context/contract-delta.ts` computes the value via the ported `repo/cross-class-peer-trigger.ts` and emits `cross_class_peer_trigger: active|inactive` in the context block; `SKILL.md` instructs the model to read that value (omit `## 5.` entirely on `inactive`, write `### Backend:` blocks on `active`). This removes the model's dependency on a shell helper and keeps §5 applicability deterministic.

_Alternative considered:_ add a model-invoked `overmind cross-class-peer-trigger` verb mirroring the old helper command. Rejected — it pushes a deterministic value back through the model and reintroduces a model-run side command the context can supply directly; the context block is the single source of the value.

_Note:_ the shared `check_cross_class_peer_trigger.sh` shell helper is **retained** (un-migrated stack-blueprint and common-contract steps still call it); only its logic is duplicated into TS for the migrated step. The two stay in lockstep until those steps migrate.

### D5 — Pending sibling contract deltas are collected in `context`; the committed-sibling lister is ported (not deleted)
`context/contract-delta.ts` reuses a ported `repo/list-committed-sibling-features.ts` (committed siblings = sibling feature folders under the project root that contain `implementation_plan.md`, byte-for-byte parity with `list_committed_sibling_features.sh`) to find each committed sibling's `feature_contract_delta.md`, and emits the present ones as read-only "pending contract delta source" inputs. The rule's instruction stands: the model reports overlaps against these pending claims but does not resolve them here. The shared shell lib is **retained** for its un-migrated callers (prerequisite-gaps, surface-map); only its logic is ported to TS for this step.

### D6 — Repo sync stays in the `sync` verb (D7 boundary); `context` only reads branch state
The old script synced ready repos to their default branch inside `collect_ready_repo_paths` (a `git pull --rebase` write) before building the prompt. Per the settled D7 repo-sync boundary, writes live in the shell-invoked `overmind sync` verb (called before the model session), and `context` only reads (`checkRepoBranchState`). This change adds `sync/contract-delta.ts` (reusing `collectReadyRepoPaths` + `syncRepoToDefaultBranch`, mirroring `sync/repo-br-scan.ts`) and registers `contract-delta` in the sync registry; `context/contract-delta.ts` runs `checkRepoBranchState` per ready repo and blocks (exit `2`, verbatim) on wrong-branch/dirty just as `repo-br-scan` context does. The phase-6 e2e launcher runs `overmind sync contract-delta` before launching the skill, exactly as phase 4.1 runs `overmind sync repo-br-scan`.

### D7 — All four read-only input classes are protected by a deterministic post-run guard
The old script's `ensure_readonly_inputs_unchanged` snapshotted **and** `cmp`-asserted every read-only input: `feature_br_summary.md`, `requirements_ears.md`, `common_contract_definition.md`, and each pending sibling `feature_contract_delta.md`. The context allowed-write list (only `feature_contract_delta.md`) and the `SKILL.md` rule are advisory text the model can ignore, so they are **not** a substitute for that guard. The transitional phase-6 e2e launcher (`run_contract_delta_skill`) re-adds the deterministic check. `context/contract-delta.ts` is the single source of truth for the protected set and emits one stable `- read_only_input: <workspace-relative-path>` entry for every resolved input. After repo sync, the launcher invokes that context command once for guard setup, parses exactly those entries, snapshots those files before launching the skill, and `cmp`-asserts each byte-unchanged after. It does not independently run `list_committed_sibling_features.sh` or reconstruct the base set. This prevents sibling-discovery drift between the guard and model context. The launcher also re-asserts that the required output `feature_contract_delta.md` was produced. This is deterministic mechanics, not skill-owned semantic logic (Ownership Rule 2/12). The allowed-write list + `SKILL.md` rule stay as defense-in-depth. When sequencing later moves to the TS orchestrator, the guard moves with it.

**The comparison runs on every exit path.** The `cmp` sweep MUST be evaluated before the launcher returns the model's exit code, including when the model session exits non-zero — otherwise a model that both corrupts a read-only input and fails would early-return the model's exit code and leave the corrupted input undetected (the exact gap in the shipped phase 5.1 launcher; see D10). When a read-only input drifted, the launcher `die`s with the read-only-corruption error even if the model also exited non-zero: a non-zero model exit is a normal restartable failure that the restart guidance handles, but a corrupted read-only input is a silent data-integrity failure a restart will not fix and that flows into every downstream step, so it is the more urgent signal and must not be masked.

The gate and context are not the right home for this check: the gate runs only against the output artifact, and context runs *before* the model so it cannot observe post-run mutation. The post-run launcher is the faithful 1:1 port.

### D8 — Two terminal final-response lines and the §5 authoring rules live only in `SKILL.md`
`feature_contract_delta.sh`/`feature_contract_delta_rule.md` defined two exact terminal lines: the success line (`Feature contract delta phase is finished. Nothing else to do now; press Ctrl-C so orchestrator can start the next phase`) and the gate-infeasibility line (`feature contract delta gate cannot pass with current EARS/common-contract inputs. Please provide instructions what to do, or adjust requirements and rerun this phase`). Both move verbatim into `SKILL.md` and nowhere else (not the e2e launcher prompt). The §5 authoring rules (mirror `transport_protocol`/`schema_format` from `common_contract_definition.md` §7 verbatim by default, record concrete values when the feature defines/refines them, use the literal `<to be defined during first feature implementation plan>` placeholder otherwise, no §5 state machine) also move verbatim into `SKILL.md`, gated on the context-supplied `cross_class_peer_trigger` value.

### D9 — e2e phase 6 mirrors the phase 4.1 (pre-sync) + phase 5.1 (read-only guard) patterns
Phase 6 follows a shape combining the two existing launchers: like phase 4.1's `run_repo_br_scan_skill` it runs `overmind sync contract-delta` before the session and preflight-checks the installed skill + CLI; like phase 5.1's `run_ears_review_skill` it snapshots read-only inputs and `cmp`-asserts them after. Immediately after sync, the launcher invokes `overmind context contract-delta <feature-path>` to obtain the stable read-only-input manifest described in D7. Context failure stops the phase before Codex launches. This deterministic guard-setup invocation does not move semantic context ownership into the shell: the skill still tells the model to invoke the same context command as its normal workflow, and the launcher prompt remains unchanged. The launcher prompt carries no literal final lines and no gate exit-code handling (single source of truth is `SKILL.md`), runs Codex from the ASDLC runtime root, and captures the model exit code without leaking `set -e`. It binds the model via the existing `feature_contract_delta` row in `.setup/models.md`: a `CONTRACT_DELTA_MODEL_PHASE="feature_contract_delta"` constant feeds `load_model_config`, which populates `MODEL_CMD`/`MODEL_MODEL`/`MODEL_ARGS`; the launcher then asserts `MODEL_CMD=codex` and builds the Codex command from the loaded model id and args rather than hardcoding them or inheriting stale globals from a prior phase. The row is reused (not added or renamed) and stays staged in `.setup/models.md` because the phase still runs, now through the skill. The `feature_contract_delta.sh` entry is removed from the phase-6 `phase_scripts` list.

### D10 — Retrofit the same every-exit-path guard onto the shipped phase 5.1 ears-review launcher
The shipped `run_ears_review_skill` (CRP-134) snapshots `feature_br_summary.md` and `cmp`-asserts it, but it early-returns the model's exit code on a non-zero model run *before* reaching the comparison — so a model that both corrupts the read-only BR and exits non-zero bypasses the guard, silently corrupting the approved BR for every downstream step. This is the same defect D7 prevents for contract-delta. Because both launchers live in the same `project_add_feature_e2e.sh`, share the read-only `feature_br_summary.md` input, and share the snapshot/`cmp` shape, this change applies the identical "comparison runs on every exit path, before returning the model rc; read-only-corruption error wins on double-failure" fix to `run_ears_review_skill` in the same edit. The fix is limited to the guard ordering — step 5.1's skill, gate, context, prompt, and final-response lines are unchanged. Doing it here (rather than a separate CRP) avoids shipping two adjacent launchers with inconsistent guard behavior and lets one e2e test fixture cover both phases, mirroring the CRP-134 D8 precedent that retrofitted the read-only-BR guard onto the phase 5 launcher.

_Alternative considered:_ leave the phase 5.1 launcher for a dedicated follow-up CRP. Rejected — the fix is a few lines mirroring the contract-delta guard, the gap is live in the merged code and silently corrupts the BR, and splitting it risks the phase 5.1 launcher staying exposed indefinitely.

### D11 — Contract-delta template and golden example are deployed only as skill assets
The migrated skill resolves `assets/feature_contract_delta_TEMPLATE.md` and `assets/feature_contract_delta_GOLDEN_EXAMPLE.md`; no retained runtime path reads `.templates/feature_contract_delta_TEMPLATE.md` or `.golden_examples/feature_contract_delta_GOLDEN_EXAMPLE.md`. Those names are therefore removed from `STAGED_TEMPLATE_FILES` and `STAGED_GOLDEN_EXAMPLE_FILES`. Fresh setup omits the flat copies, while update mode removes pre-migration copies through the existing exact-manifest cleanup for `.templates/` and `.golden_examples/`. No additional obsolete-file cleanup path is needed.

### D12 — Deployability guidance references only existing scripts
The repository-local `overmind-step-deployability` meta-skill previously named the deleted `overmind/scripts/feature_contract_delta.sh` as a reference deployable step. Remove that stale entry and retain `overmind/scripts/feature_implementation_plan.sh` as the valid legacy-script example. This is a documentation-only correction to ignored local skill guidance; it does not change runtime staging or step behavior.

## Risks / Trade-offs

- **[awk → TS branch/field drift]** → mitigated by porting the section/meta/handoff rules and the `delta_needed` true/false branch (Delta-block field set vs. the `no_contract_delta_required` line) exactly, including key/value normalization, and pinning every edge case from `check_feature_contract_delta_quality_tests.sh` in TS tests before deleting the bash. The §5 gate-exemption is pinned by a test asserting a §5-bearing and a §5-absent artifact both pass on identical sections 1–4.
- **[Shared-helper divergence]** → the cross-class trigger and sibling lister are duplicated into TS while the shell helpers stay live for un-migrated callers; mitigated by porting each one-for-one against its existing shell test (`check_cross_class_peer_trigger_tests.sh`, `list_committed_sibling_features_tests.sh`) and adding TS tests covering the same cases, so the two implementations stay in lockstep until the shell callers migrate.
- **[Repo-sync boundary regression]** → moving the `git pull --rebase` write out of context into the `sync` verb risks the launcher forgetting to pre-sync; mitigated by mirroring the phase 4.1 launcher exactly and an e2e test asserting phase 6 runs `overmind sync contract-delta` before the session and that context only branch-checks (no write).
- **[Read-only-input boundary regression]** → the old script protected four input classes, including dynamically-discovered sibling deltas; dropping that for prompt/context text alone would turn a guaranteed deterministic failure into silent corruption that flows downstream. Mitigated by re-adding the deterministic snapshot+`cmp` guard over all four classes in the phase-6 launcher (D7), keeping the single-target allowed-write list + `SKILL.md` rule as defense-in-depth, and an e2e test that fails the phase when the stubbed model mutates any read-only input (including a sibling delta).
- **[Losing a terminal final line / the §5 authoring rules]** → mitigated by a parity sweep table (old script+rule+helper vs SKILL.md+context+gate+sync) that pins both exact lines and the §5 authoring rules, and a test asserting the final lines appear only in `SKILL.md` (not the e2e launcher).

## Migration Plan

1. Land the shared ports + TS core: `repo/list-committed-sibling-features.ts`, `repo/cross-class-peer-trigger.ts`, `validate/contract-delta.ts`, `context/contract-delta.ts`, `sync/contract-delta.ts`, registries in `cli/run.ts`, with tests — `npm test --workspace packages/asdlc-coordinator` green.
2. Land the `overmind-contract-delta` skill payload + installer wiring (`init.ts` sixth skill) — `npm test --workspace packages/installer` green.
3. Land setup staging + e2e phase-6 rewiring (pre-sync + every-exit-path read-only guard) and the phase 5.1 guard-ordering retrofit (D10); update shell suites — `project_setup_asdlc_tests.sh` / `project_setup_update_project_tests.sh` / `project_add_feature_e2e_tests.sh` green.
4. Parity sweep (old script+rule+helper vs SKILL.md+context+gate+sync), then delete the migrated step-6 bash, rule, helper, and the two old shell test files (leaving the shared cross-class/sibling helpers and their tests in place); update `CLAUDE.md`/`AGENTS.md`, README/QUICKRUN, and the migration overview (mark step 6 done).
5. `git diff --check`; `openspec validate crp-135-migrate-feature-contract-delta --strict`.

Rollback: revert the change; nothing depends on the new `contract-delta` gate/context/sync yet, and the migrated and downstream steps are untouched.
