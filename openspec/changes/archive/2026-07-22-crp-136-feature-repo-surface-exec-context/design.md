## Context

Step 7 of the Overmind init/feature pipeline is "Surface map BE/FE". Today it is one bash script driven by the e2e orchestrator's phase-7 class loop (`project_add_feature_e2e.sh` `run_phase7_loop` → `feature_repo_surface_and_exec_context.sh`):

- `feature_repo_surface_and_exec_context.sh` (+ `feature_repo_surface_and_exec_context_rule.md` shared rule + two per-class quality helpers) — a **per-class deterministic single-target model step**: it resolves the feature path and `projects/<id>` root; reads `meta_info.project_classes`; for each active class (`backend`/`frontend`/`mobile`) builds the set of analyzable targets (a class with a **ready** `class_repo_paths` entry → its resolved repo path; or, when not ready, a class whose `project_stack_blueprint_<class>.md` exists → a blueprint-fallback target with no repo path); **interactively selects** one target; **syncs** the selected ready repo to its default branch; binds per-class assets (backend → `_be_` template/golden/helper and output `project_surface_struct_resp_map_backend.md`; frontend → `_fe_` and `…_frontend.md`; mobile → `_fe_` and `…_mobile.md`); snapshots the read-only inputs (`init_progress_definition.yaml`, `requirements_ears.md`, `feature_contract_delta.md`, plus the optional `project_stack_blueprint_<class>.md` and every committed sibling `implementation_plan.md`); launches a Codex session; asserts the per-class output was produced and `cmp`-asserts none of the read-only inputs changed. The two helpers validate the per-class surface map structure (title, four sections, nine meta keys, three scope keys, seven layer subsections each with five fields, the `3.8 Another Layer(s)` subsection, eight surface subsections each with eight fields, ≥1 applicable surface) with awk; the BE and FE helpers differ only in the title, the section-4 heading, the layer/surface names, and the `project_classes` assertion.

The phase-7 loop (`run_phase7_loop`) already owns the per-class iteration: it computes active classes from the definition, marks a class **completed** when its `project_surface_struct_resp_map_<class>.md` exists and **pending** otherwise, and on "Analyze one class now" runs the bash script (which itself prompts for the class). Class scope/state (`ready`/`deferred`) is operator-set via repo-attach, unchanged here.

This change is the eighth in the md+sh → skill+TS migration, after CRP-129→135. The constraints from `design_docs/to_skills_migration/*` apply: clean break (no dual bash+TS path), the model owns the artifact loop, JS/TS owns deterministic mechanics, the gate is model-invoked with the `0/1/2` contract, repo writes (sync) live in the shell-invoked `sync` verb (D7 boundary), and skills install only to the supported `.codex`/`.claude` runner targets sharing the single `.overmind/overmind.js` CLI. The design overview names this skill `overmind-surface-map`; gate name = skill name minus the `overmind-` prefix → `surface-map`. Critically, the settled decision **"Type & per-class branching → resolved by the orchestrator, not SKILL.md conditionals; per-class runs one invocation per ready class (passes `class=be|fe`), step-skills stay single-case and parameterized"** governs the whole shape of this migration.

## Goals / Non-Goals

**Goals:**
- Migrate step 7 to the per-class `overmind-surface-map` skill backed by a `--class`-parameterized `surface-map` gate + context builder + `sync` verb, with behavior parity to the rule + the two helpers + the orchestrator.
- Keep the skill/gate/context/sync **single-class and parameterized**; keep class **selection** in the orchestrator's phase-7 loop (per the settled per-class branching decision).
- Preserve the per-class binding (BE vs FE/mobile assets, output filename, title/section-4/layer/surface sets, `project_classes` assertion), the blueprint-fallback-when-no-ready-repo behavior, the read-only boundary over all five input classes, and the two track-parameterized final-response lines.
- Reuse the CRP-131 repo machinery and the CRP-135 `repo/list-committed-sibling-features.ts` and read-only-manifest / every-exit-path-guard patterns rather than re-inventing them.
- Fold the installer/setup/e2e wiring into this same change so no follow-up is needed.
- Port both BE/FE bash test suites to TS-runner + shell-suite coverage, then delete the migrated step-7 bash, rule, both helpers, and old tests (clean break).

**Non-Goals:**
- The cross-step TS orchestrator / state machine; the e2e shell phase-7 loop stays the transitional per-class sequencer.
- Migrating `project_add_feature_e2e.sh` / `project_setup_first_init_machine.sh` themselves to TS.
- `.github`/`.agents` runner fan-out (still deferred).
- Step 7.1 (MCP placeholder enrichment) and all later steps; migrating the un-migrated callers of the shared repo/cross-class/sibling helpers.

## Decisions

### D1 — One parameterized skill + a `--class` option on gate/context/sync, no capture and no readiness verb
Step 7 has no user-captured input and no state-flip of its own. Following the settled per-class branching decision, it is **one** skill (`overmind-surface-map`) and **one** of each verb, each taking `--class <backend|frontend|mobile>`. The skill is single-class; the orchestrator invokes it once per selected class. This avoids per-class skill duplication and SKILL.md `if backend … else frontend` conditionals, keeping the class fan-out in the orchestrator where project-type/class logic is already resolved. No `capture`, no `readiness`.

_Alternative considered:_ three skills (`overmind-surface-map-be/fe/mb`) or SKILL.md class conditionals. Rejected — it contradicts the settled "step-skills stay single-case and parameterized" decision and triples the asset/installer surface.

### D2 — `--class` is a first-class CLI option on `gate`/`context`/`sync`, validated centrally
`cli/run.ts` currently parses options only for `capture`. This change adds a small option parser for `gate`/`context`/`sync surface-map` that reads `--class <value>`, validates `value ∈ {backend, frontend, mobile}`, and exits `2` with a usage error on a missing/unknown class. The verb handlers pass the class through to the registered builder/validator/syncer. Other steps' verbs ignore the option (they take no class), so the CLI shape stays backward-compatible.

### D3 — One validator with two per-class configs; mobile reuses the frontend config
`check_…_be_quality.sh` and `…_fe_quality.sh` are byte-for-byte identical except for a per-class config block (title string, section-4 heading, the seven `### 3.x` layer names, the eight `### 4.x` surface names, and the `project_classes` membership assertion — backend requires `backend`; frontend/mobile requires `frontend` or `mobile`). `validate/surface-map.ts` ports the shared awk engine once (placeholder detection incl. `[OPTIONAL…]`, section/meta/scope/layer/surface parsing, the per-field `is_unfilled` checks, `project_type_code ∈ {A,B,C}`, `last_updated` `YYYY-MM-DD`, ≥1 `applicability: applicable`) and selects the config by class; `mobile` maps to the frontend config (matching the old `configure_target_bindings`). Exit codes map to the migration contract (`0` pass with the per-class pass message, `1` with the same `quality gate failed: …` strings, `2` runtime). The validator reads only the target artifact.

### D4 — Ready-repo-or-blueprint-fallback scan scope, resolved in `context`
The old `collect_surface_targets` treated a class as analyzable if it had a **ready** repo path **or** a `project_stack_blueprint_<class>.md` (blueprint fallback = "no ready repository; blueprint evidence is primary planned structural evidence"). `context/surface-map.ts` reproduces this for the requested class: resolve the class's ready repo via `collectReadyRepoPaths` (filtered to that class, with `checkRepoBranchState` → block/exit `2` verbatim on dirty/wrong-branch); if no ready repo, look for `project_stack_blueprint_<class>.md` and emit a blueprint-fallback scan scope (no repo path); if neither exists, exit `2` (the class is not analyzable). The blueprint, when used, is added to the read-only manifest. This keeps the model's scan scope deterministic and the "blueprint is planned evidence" semantics intact.

### D5 — Repo sync stays in the `sync` verb (D7 boundary); blueprint-fallback is a no-op
The old script synced the selected ready repo to its default branch before building the prompt. Per the settled D7 boundary, that write (`git pull --rebase`) lives in `overmind sync surface-map <feature> --class <class>` (reusing `collectReadyRepoPaths` + `syncRepoToDefaultBranch`, filtered to the class), called by the phase-7 launcher before the session; `context` only reads (`checkRepoBranchState`). A blueprint-fallback class (no ready repo) syncs nothing and exits `0`.

### D6 — Class selection moves from the deleted script into the orchestrator phase-7 loop
The old bash did class selection internally (`select_target_repo`, reading stdin). Per the per-class branching decision, selection is orchestrator-owned: `run_phase7_loop`'s "Analyze one class now" branch presents the **pending** classes, takes the operator's pick, and calls `run_surface_map_skill <class>`. The skill/context/gate/sync receive the chosen class via `--class`; they never prompt for it. Completion tracking (map file exists → class completed) is unchanged, so the loop naturally advances class-by-class until the operator moves forward.

### D7 — Read-only guard over all five input classes, on every exit path
The old `main` snapshotted and `cmp`-asserted every read-only input: `init_progress_definition.yaml`, `requirements_ears.md`, `feature_contract_delta.md`, the optional `project_stack_blueprint_<class>.md`, and every committed sibling `implementation_plan.md`. The phase-7 launcher (`run_surface_map_skill`) re-adds this deterministically, reusing the CRP-135 pattern: derive the read-only set from the `- read_only_input:` manifest emitted by `context surface-map --class <class>` (single source of truth), snapshot each before the session, and `cmp`-assert each byte-unchanged after — evaluated on **every** exit path, before returning the model's exit code, with the read-only-corruption error winning on a double-failure (model mutates an input and also exits non-zero). It also asserts the per-class output `project_surface_struct_resp_map_<class>.md` was produced. The context allowed-write list (that one file) and the `SKILL.md` rule stay as defense-in-depth.

### D8 — Two track-parameterized final-response lines live only in `SKILL.md`
The old script built the success/infeasibility lines from `TARGET_TRACK_LABEL` (`backend`/`frontend`/`mobile`). Both move verbatim into `SKILL.md` with the `<track>` placeholder filled from the context-supplied track label, and nowhere else (not the e2e launcher prompt). The infeasibility line is the model's escape hatch when the gate cannot pass with the current repository/blueprint evidence.

### D9 — Reuse the shared sibling lister; keep the shell libs for un-migrated callers
The in-flight evidence set (committed siblings' `implementation_plan.md`) is the same "committed sibling" notion CRP-135 ported to `repo/list-committed-sibling-features.ts`; `context/surface-map.ts` reuses it directly (no re-port). The shell `list_committed_sibling_features.sh`, `class_repo_paths.sh`, `sync_repo_to_default_branch.sh`, and `check_cross_class_peer_trigger.sh` libs stay staged because the un-migrated step 7.1 / prerequisite-gaps / later steps still call them.

### D10 — The two surface quality helpers are RETAINED (step 7.1 depends on them), not deleted
`validate/surface-map.ts` ports the BE/FE helper logic for the migrated step's gate, but the shell helpers `check_feature_repo_surface_and_exec_context_be_quality.sh` and `..._fe_quality.sh` are **not deleted and stay staged**: the un-migrated step 7.1 (`feature_surface_map_mcp_placeholder_enrichment.sh`) declares them as required files, invokes them per class, and cites them in its prompt. This is the same retain-the-shared-helper pattern CRP-135 used for `check_cross_class_peer_trigger.sh` (ported to TS for the migrated step, shell kept for un-migrated callers). The TS validator and the shell helpers stay in lockstep (both ported one-for-one from the same awk) until step 7.1 migrates. Only the migrated orchestrator script, its rule, and its two old orchestrator test suites are removed; the helpers' structural behavior is covered going forward by `validate/surface-map.ts` TS tests plus the existing step-7.1 MCP enrichment test that stubs/exercises them. (Step 7.1 uses its own `feature_surface_map_mcp_placeholder_enrichment_rule.md` and does not read the flat surface templates/golden examples, so the rule deletion and the templates/golden move-to-skill-local are safe.)

## Risks / Trade-offs

- **[awk → TS per-class config drift]** → mitigated by porting the shared engine once and the BE/FE config blocks exactly (titles, section-4 headings, the seven layer names, the eight surface names, the `project_classes` assertion, `[OPTIONAL…]` placeholder detection, `project_type_code`/`last_updated` formats, ≥1-applicable rule), and pinning every BE and FE edge case from the two `*_tests.sh` suites in TS tests before deleting the bash. A test asserts `mobile` uses the FE config and writes `…_mobile.md`.
- **[`--class` plumbing regressions]** → mitigated by a central validated option parser with `2`-exit on missing/invalid class, and CLI tests for each verb (`gate`/`context`/`sync surface-map`) covering valid classes, missing `--class`, and an unknown class.
- **[Blueprint-fallback loss]** → dropping the "no ready repo ⇒ use stack blueprint as planned evidence" path would block analysis for not-yet-attached classes; mitigated by reproducing it in `context` (ready-repo-or-blueprint, exit `2` only when neither exists) and a context test for the fallback scan scope + blueprint in the read-only manifest.
- **[Per-class read-only boundary regression]** → the old script protected five input classes incl. the dynamically-chosen blueprint and in-flight plans; dropping that for prompt/context text alone would silently corrupt inputs feeding every downstream step. Mitigated by the every-exit-path manifest-driven `cmp` guard (D7) and an e2e test that fails the phase when the stubbed model mutates any read-only input (incl. a blueprint and an in-flight plan), including the model-also-fails case.
- **[Class-selection ownership confusion]** → moving selection into the loop while the skill stays single-class could leave a gap; mitigated by an e2e test asserting the phase-7 loop prompts for a pending class, passes the right `--class` to sync/skill, marks the class completed when its map file appears, and loops to the next pending class.

## Migration Plan

1. Land the TS core: `validate/surface-map.ts` (BE+FE configs), `context/surface-map.ts` (per-class binding + blueprint fallback + manifest), `sync/surface-map.ts`, the `--class` option parser + registries in `cli/run.ts`, reusing `repo/list-committed-sibling-features.ts` — `npm test --workspace packages/asdlc-coordinator` green.
2. Land the `overmind-surface-map` skill payload (SKILL.md + four assets) + installer wiring (`init.ts` seventh skill) — `npm test --workspace packages/installer` green.
3. Land setup staging (stage skill + skill-local assets; remove migrated bash + flat surface-map assets) + phase-7 loop rewiring (select class → sync → skill with every-exit-path guard); update shell suites — `project_setup_asdlc_tests.sh` / `project_setup_update_project_tests.sh` / `project_add_feature_e2e_tests.sh` green.
4. Parity sweep (old script+rule+both helpers vs SKILL.md+context+gate+sync), then delete the migrated step-7 bash, rule, both helpers, and the two old shell test files; update `CLAUDE.md`/`AGENTS.md`, README/QUICKRUN, and the migration overview (mark step 7 done).
5. `git diff --check`; `openspec validate crp-136-feature-repo-surface-exec-context --strict`.

Rollback: revert the change; nothing depends on the new `surface-map` gate/context/sync yet, and the migrated and downstream steps are untouched.
