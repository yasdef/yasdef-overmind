## Why

The E2E orchestrator migration (`design_docs/e2e_orchestrator_migration/`) replaces the bash e2e/scanner with a TypeScript functional core. Slice 0 (`crp-144`) landed the toolchain baseline and Slice 1 (`crp-145`) landed the pure `workspace/` + `sequencing/` foundation; this change is Slice 2 — **everything needed to launch one model session correctly, as reusable modules** (`04_migration_plan.md ## Slice 2 — Runner config + agent runner + guards`, contracts in `03_target_architecture.md ## Key contracts`).

Slice 1 defined the step catalog as declarative data whose `Action[]` metadata is typed but **not executed**. Slice 2 makes those actions executable: it collapses the shell e2e's 13 near-identical `run_*_skill` launchers, `load_model_config` awk, 13 `build_*_prompt` heredocs, per-phase `MODEL_CMD=codex` assertions, and hand-copied `mktemp`/`cmp` guard blocks into **one** data-driven executor with a typed config loader, one prompt builder, one `AgentRunner` port, and one guard module. As part of that cutover, Slice 2 also corrects the catalog's guard metadata to match the shell's real ownership boundaries: early sessions stay unguarded, steps 5/5.1 use `mustExistUnchanged`, steps 6/7/8/8.x use `fromContext`, and step 7.1 uses `preserveExistence` with runtime-correct paths. No orchestrator loop yet (Slice 3) — these are the reusable building blocks the loop composes.

## What Changes

- Add **`config/`**: a typed loader + validation for the **unchanged** `.setup/models.md` pipe-table (decision 5 in `03_target_architecture.md ## Decisions`: format kept, awk replaced). Parses each `phase | command | model | args...` row into a typed per-phase `{ command, model, args[] }` structure; `command` is validated against the registered agent adapters (only `codex` initially — preserving today's thirteen assertions as one rule). Load problems surface as `Diagnostic` values in the loader's typed result — actionable at startup, renderable by CLI and extension alike (map row 6). No workspace migration.
- Add **`runner/` prompt builder**: one pure prompt builder over a `StepDefinition` session action + runtime bindings, replacing the 13 `build_*_prompt` heredocs. Skill-owned final-response lines stay out, exactly as today (map row 7).
- Add **`runner/` AgentRunner**: the `AgentRunner` port plus the `CodexAgentRunner` adapter (spawn from runtime root, inherit stdio, return exit code). Other runners become additional adapters later; config keeps `codex` as the only allowed command for now (map row 8).
- Add **`runner/` guards**: a snapshot/verify module driven by the typed `readOnlyGuards` union — `fromContext` (protected set = the context result's read-only inputs; steps 6/7/8/8.x), `mustExistUnchanged` (byte-identical named files; steps 5/5.1), `preserveExistence` (unchanged if present, absent must stay absent, never created/deleted/replaced; step 7.1) — plus the `requiredOutputs` existence assertion after each session, with **empty-list-is-legal** semantics (7.1 mutates in place, asserts nothing new) (map row 11).
- **Correct `sequencing/step-catalog.ts` guard metadata** so the executor can trust catalog data instead of step-id branches: remove the incorrect Slice 1 default that applied `fromContext` to every session, assign `mustExistUnchanged` to steps 5/5.1, assign `fromContext` only to steps 6/7/8/8.x, and fix step 7.1's `preserveExistence` paths to the runtime-relative `.setup/external_sources.yaml` and project `init_progress_definition.yaml`.
- Add **`runner/` generic executor**: `executeStep(stepDef, bindings)` iterates a step's declared `Action[]` with **no per-step branches** — for a session: evaluate `runIf` (skip with notice) → sync (in-process) → context (in-process, yields the read-only inputs) → snapshot guards → build prompt → `AgentRunner.run()` → verify guards + required outputs; for a deterministic action: call the named coordinator function. Any action failure stops the step with a typed result. Session bindings cover the feature path and a single class (the class-list binding is Slice 4). Preserves the D7 boundary (repo-mutating sync in the operator process before the session; context builders read-only) as in-process function calls (map row 5 executor part; rows 9/10/12 reused in-process).
- **Extend `context/` (additive contract change)**: add an optional structured `readOnlyInputs: string[]` to `ContextResult` and populate it in the `fromContext` builders from the same paths the existing `- read_only_input:` text lines render. The Slice 2 `fromContext` guards depend on `ContextResult.readOnlyInputs`; today those paths live only inside `ContextResult.text` (the shell scraped them with `sed`). Strictly additive — `.text` stays byte-unchanged, so the standalone `context` CLI verb is unaffected — honoring the "zero output parsing" claim (`02_responsibility_translation_map.md` row 10; design D5).
- Add **`interaction/`**: the `InteractionPort` types (confirm / select / input as typed messages) plus the TTY adapter with today's prompt wording and semantics. Needed by Slice 3; specified here so the runner result / interaction shapes co-evolve (map row 14, partial — port + adapter only).
- **Tests (TS):** config validation (well-formed read + missing/malformed row degradation, non-`codex` command rejected); prompt-content parity against the 13 heredocs (skill name, runtime bindings, exact CLI commands, no final-response lines); stub `AgentRunner` adapter; guard tests covering all three modes — explicitly the 7.1 shape (absent-stays-absent, exists-unchanged, empty `requiredOutputs`) alongside `fromContext` and `mustExistUnchanged`; executor tests over the stub agent (runIf skip, action-sequence ordering, sync/context/guard ordering, failure stops the step).
- **Deletes nothing** — the shell e2e still reads `.setup/models.md` and runs its own launchers until Slice 3 (`overmind run`) replaces it. No transitional shell edits in this slice.
- **No new runtime dependencies:** `asdlc-coordinator`'s `dependencies` list stays empty (zero-runtime-dependency rule); Node's built-in test runner only.

## Capabilities

### New Capabilities
- `runner-config`: the typed `config/` loader + validation for the unchanged `.setup/models.md` pipe-table — per-phase `{ command, model, args[] }`, `codex`-only command rule, load problems as `Diagnostic` values (map row 6).
- `context-read-only-inputs`: the additive `ContextResult.readOnlyInputs` typed field populated in the `fromContext` builders (byte-unchanged `.text`; field matches the rendered `- read_only_input:` lines) — the typed source the `fromContext` guards and executor consume instead of scraping stdout (map row 10; design D5).
- `session-prompt-builder`: the single pure prompt builder over a `StepDefinition` session action + runtime bindings, replacing the 13 heredocs with skill-owned final-response lines kept out (map row 7).
- `agent-runner`: the `AgentRunner` port and the `CodexAgentRunner` adapter (spawn from runtime root, inherit stdio, exit-code result) — the seam that keeps codex swappable and the extension's terminal-hosted runner a later adapter (map row 8).
- `session-guards`: the snapshot/verify module for the `readOnlyGuards` union (`fromContext` / `mustExistUnchanged` / `preserveExistence`) plus the `requiredOutputs` existence assertion with empty-list-is-legal semantics — one implementation, applied uniformly, 7.1 is data not a branch (map row 11).
- `step-executor`: the generic `executeStep(stepDef, bindings)` action-sequence executor — no per-step branches, `runIf` evaluation, in-process sync/context/readiness reuse, the D7 ordering, and the typed per-action result (map row 5 executor part; reuses existing `sync/`, `context/`, `readiness/` in-process).
- `interaction-port`: the `InteractionPort` types (confirm / select / input) and the TTY adapter preserving today's wording — the seam Slice 3's loop and the extension's webview forms both implement (map row 14, partial).

## Impact

- **New (`packages/asdlc-coordinator/src/`):** `config/` (runner-config loader), `runner/` (prompt builder · guards · `AgentRunner` port + `CodexAgentRunner` adapter · generic `executeStep`), `interaction/` (port types + TTY adapter). Possibly small typed-result additions in `types/` for the runner/executor/interaction shapes.
- **Extended — additive contract change (`context/`, `types/`):** `ContextResult` gains an optional structured `readOnlyInputs: string[]`, populated in the `fromContext` builders (contract-delta, surface-map, technical-requirements, implementation-slices, prerequisite-gaps, implementation-plan, plan-semantic-review, surface-map-enrich) from the same paths the existing `- read_only_input:` text lines render. This is required because the Slice 2 `fromContext` guards depend on `ContextResult.readOnlyInputs`; today those paths exist only inside `ContextResult.text` (the shell scraped them with `sed`). The change is strictly additive — `.text` stays byte-unchanged, so the standalone `context` CLI verb and any other consumer are unaffected (design D5, tasks §3).
- **Extended (`sequencing/`):** `step-catalog.ts` remains the declarative source of action sequencing, but its guard metadata is corrected in this slice to match shell ownership and runtime path reality: no default `fromContext` on every session, explicit `mustExistUnchanged` for steps 5/5.1, explicit `fromContext` only for steps 6/7/8/8.x, and corrected 7.1 `preserveExistence` paths.
- **Reused in-process (no change):** `sync/` (`repo-br-scan`, `contract-delta`, `surface-map`, `prerequisite-gaps`) and `readiness/` (`br-clarification`).
- **New tests (TS):** config validation; prompt-content parity vs the 13 heredocs; stub `AgentRunner`; guard tests for all three modes incl. the 7.1 shape and empty `requiredOutputs`; executor tests (runIf skip, ordering, failure-stops-step) over the stub agent.
- **No shell edits, no deletions:** the shell e2e keeps reading `.setup/models.md` and running its launchers until Slice 3; `.setup/models.md` format is unchanged (no workspace migration).
- **Verification:** green local `npm run verify` (typecheck → lint → format-check → build → test across TS workspaces and the surviving `tests/ai_scripts/*.sh` suites) is the completion criterion; `asdlc-coordinator` runtime `dependencies` stays empty.
- **Out of scope:** the orchestrator loop / `overmind run` and the scaffold primitive (Slice 3), the class-list session binding and `overmind project reconcile` (Slice 4), `state/` and `git/` modules (Slices 3–4), the VS Code extension implementation (Slice 5). No changes to skill bodies, gates, or artifact formats.
