## Context

CRP-129 established the migration pattern: a per-step bash trio (`md` rule + `sh` orchestrator + `sh` gate helper) dissolves into an agent **skill** (rule inlined into `SKILL.md`) plus deterministic `overmind capture/context/gate <step>` mechanics in the shared `asdlc-coordinator` core. CRP-130 made installation fan the packaged skill into the supported local runners (`.codex/skills/`, `.claude/skills/`) while `.overmind/overmind.js` stays the single shared CLI. The most recent orchestrator change (commit `1e1305b`, "fix 4.2 run e2e issue") made `project_add_feature_e2e.sh` **actively drive** the `overmind-task-to-br` skill in phase 4.1 through a Codex session (`run_task_to_br_skill` + `build_task_to_br_prompt` + `load_model_config`), running `feature_scan_repo_for_br.sh` first and the task-to-BR skill second.

This migration follows the canonical playbook `design_docs/to_skills_migration/step_by_step_migration_for_particular_step.md` (Inputs, ownership rules, JS primitive layout, skill structure, the §5 instruction-parity table, install/setup staging, transitional e2e wiring, clean-break deletion, tests, and Definition of Done). The Section 0 inputs for this step: `STEP_ID=4.1`, `STEP_KEY=repo-br-scan`, `SKILL_NAME=overmind-repo-br-scan`, `MODEL_PHASE=repo_analyse`, `OLD_SCRIPT=overmind/scripts/feature_scan_repo_for_br.sh`, `OLD_RULE=overmind/rules/repo_br_scan_rule.md`, `OLD_HELPER=overmind/scripts/helper/check_business_context_filled_from_repo.sh`, `OLD_TESTS=tests/ai_scripts/init_scan_repo_for_br_tests.sh`, `TARGET_ARTIFACTS=feature_br_summary.md` (only `## 1. Document Meta` `last_updated`/`source_type` and `## 13. Existing-System Context`), `READ_ONLY_INPUTS=init_progress_definition.yaml` + the ready class repositories, `CAPTURE_ARTIFACTS=none`.

This change migrates the **repo-scan** half of step 4.1. Current assets being replaced:
- `overmind/scripts/feature_scan_repo_for_br.sh` — orchestrator. Beyond hand-off it: resolves the feature path + target `feature_br_summary.md`, resolves `init_progress_definition.yaml` from the nearest ancestor, collects **ready** class repo paths via `class_repo_paths_collect_ready_paths`, **syncs each ready repo to its default branch** via `sync_repo_to_default_branch.sh` (D7), and builds the model prompt (repos-to-scan list, rule reference, gate command). No ready classes ⇒ explicit no-op.
- `overmind/rules/repo_br_scan_rule.md` — repo-scan enrichment rule (evidence discipline, the strict `## 13` repository-block format contract, the Business Context Completeness Gate loop).
- `overmind/scripts/helper/check_business_context_filled_from_repo.sh` — gate over `feature_br_summary.md`: `## 1. Document Meta` (`source_type`/`last_updated` present, filled, `YYYY-MM-DD`) and `## 13. Existing-System Context` (present, ≥1 field, no `[UNFILLED]`).
- `tests/ai_scripts/init_scan_repo_for_br_tests.sh` — bash suite (argument/staging guards, ready-class gating, no-op, and the D7 git-sync block scenarios).

## Goals / Non-Goals

**Goals:**
- Migrate step 4.1 repo-scan to the `repo-br-scan` step in `asdlc-coordinator` (`validate/` gate parity, `context/` builder with D7 repo sync) plus the `overmind-repo-br-scan` skill, with ported tests.
- Reuse the CRP-129 trichotomy and CLI; reuse the CRP-130 runner-install pattern, generalized to install **both** skills so this change needs no follow-up.
- Rewire phase 4.1 of the e2e orchestrator to drive the new skill via a Codex session, preserving "scan-then-task-to-BR" order and the no-ready-class no-op.
- Clean-break delete the four replaced bash files while retaining the still-shared `common_libs/*`.

**Non-Goals:**
- The cross-step TS orchestrator/state-machine, and migrating `project_add_feature_e2e.sh` / `project_setup_first_init_machine.sh` themselves to TypeScript (the bash orchestrator/installer keep driving skills for now).
- `.github/.agents` runner fan-out; standalone-binary compilation; other pipeline steps.
- Reworking the strict `## 13` repository-block format contract: it stays model-enforced (inlined `SKILL.md` prose), matching the bash gate which never validated block shape — only `[UNFILLED]` completeness.

## Decisions

- **Repo-scan becomes its own skill `overmind-repo-br-scan`, not folded into `overmind-task-to-br`.** The migration overview table lumps step 4.1 as "Scan repo + task-to-BR", but the two sub-phases have distinct rules (`repo_br_scan_rule.md` vs `task_to_br_rule.md`), distinct gates (`## 13` business-context completeness vs `## 1`/`2.1`/`3.1`/`6`/`7` BR structure), distinct editable surfaces, and distinct triggers (repo evidence vs captured user input). One skill ⇒ one rule ⇒ one gate keeps `SKILL.md` single-case, consistent with how CRP-129 already created `overmind-task-to-br` for only the task-to-BR rule. The design overview's step-4.1 row is updated to show both skills.
- **The D7 repo-sync precondition lives inside `context/repo-br-scan.ts`, not a new CLI verb.** The bash orchestrator synced ready repos as part of assembling usable context (`collect_usable_repo_paths` ran `sync_repo_to_default_branch.sh` before `build_prompt`). Keeping it in `context` preserves the capture/context/gate trichotomy and keeps the sync deterministic (host-owned, not model prose). Consequence: `context repo-br-scan` is **not** side-effect-free — it runs `git pull --rebase` on each ready repo. A D7 block (non-default branch, dirty tree, missing/ambiguous default, no upstream, rebase failure) makes `context` exit `2` with the original `BLOCKED: ... (D7) — ...` message, so the model stops and asks the user — matching the bash behavior of refusing to scan and not touching the artifact. *Alternative considered:* a separate `overmind sync-repos` verb (rejected: expands the CLI contract for one step; the model would have to sequence sync vs context itself).
- **No `capture` for `repo-br-scan`.** There is no user-provided input to capture; the input is the pre-existing `feature_br_summary.md` plus the ready repositories. The CLI keeps `capture` unregistered for this step (unknown-step ⇒ exit `2`), so only `context` and `gate` are wired.
- **Gate parity is `awk`-for-`awk`: completeness only.** `validate/repo-br-scan.ts` reproduces exactly what `check_business_context_filled_from_repo.sh` checked — `## 1. Document Meta` keys and `## 13. Existing-System Context` field completeness — and nothing more. The strict `## 13` per-repository block format contract stays in the inlined rule (model-enforced). Each distinct failure emits one actionable `missing: ...` line.
- **Shared repo logic is ported to TS but the bash `common_libs/*` are retained.** `class_repo_paths.sh` and `sync_repo_to_default_branch.sh` are consumed by many still-bash steps (contract-delta, prerequisite-gaps, surface-map, project setup, …). The TS `repo/` module is a parallel implementation for the migrated step; the bash libs are deleted only when their last bash consumer migrates. This is the same "drain bash one step at a time" stance as CRP-129.
- **Installer + setup are generalized to multi-skill in this change (avoids a CRP-130 repeat).** CRP-130 hardcoded a single `SKILL_NAME`. Here `installer/src/init.ts` and the setup script's `stage_runner_skills` iterate a skills list (`overmind-task-to-br`, `overmind-repo-br-scan`) × supported runners. This is the smallest generalization that lets future migrated skills register without another installer change.
- **e2e phase 4.1 mirrors the `run_task_to_br_skill` pattern.** A new `run_repo_br_scan_skill` (+ `build_repo_br_scan_prompt`, reusing `load_model_config` with the existing `repo_analyse` model phase) drives the `overmind-repo-br-scan` skill via Codex. Phase 4.1 runs the scan skill before the task-to-BR skill when a class is ready, and skips the scan (no-op) when none is ready — preserving the order and gating the deleted bash script enforced.
- **The completion / final response line lives only in `SKILL.md`; the orchestrator prompt does not duplicate it.** Skills are launched by the orchestrator, never started independently, so every model-facing semantic instruction the skill owns — the final response line `Repo scan phase to enrich BR is finished. Nothing else to do now; press Ctrl-C so orchestrator can start the next phase`, gate exit-code handling, and the editable-write surface — lives in `SKILL.md` as the single source of truth. `build_repo_br_scan_prompt` stays a thin launcher: runtime path bindings, "load and follow the `overmind-repo-br-scan` skill", and the exact `context`/`gate` commands. It MUST NOT restate the final response line or any other skill-owned instruction. The orchestrator still advances the phase on the skill-emitted line, but the line is authored exactly once. This refines playbook §7.5 ("final response line if required"): it is required only where it is *owned* — `SKILL.md` — not copied into the launcher, so the two can never drift. `build_task_to_br_prompt` already follows this single-source rule (its final line lives only in `overmind-task-to-br/SKILL.md`); `build_repo_br_scan_prompt` is authored to the same shape.
- **Instruction parity is verified with the playbook §5 table before deletion.** A row-by-row comparison of `OLD_SCRIPT` + `OLD_RULE` against `SKILL.md` + context/gate must show no `missing` row — in particular scrubbing stale ordering (e.g. any "before/after repo scan" phrasing that no longer matches the sequence diagram), preserving evidence-discipline / no-invention / contradiction-handling rules, the editable-surface allow-list (`## 1` meta + `## 13` only), and the gate-after-every-write instruction.

## Risks / Trade-offs

- **Behavior drift in the bash→TS port (gate + D7 sync + path collection)** → Port `check_business_context_filled_from_repo.sh` checks one-for-one (each failure covered by a fixture); reproduce `class_repo_paths_collect_ready_paths` (ready filter, empty-path rejection, dir existence, dedupe) and `sync_repo_to_default_branch.sh` (every `BLOCKED: ... (D7)` branch, including rebase-abort and linked-worktree git-dir handling) with git-fixture tests ported from `init_scan_repo_for_br_tests.sh`. Do this before deleting the bash files.
- **`context` now mutates repos** → Document the side effect in `SKILL.md` and the context block; on any D7 block, exit `2` and write nothing (parity with bash). The model never edits the artifact when context fails.
- **Two changes (CRP-130, CRP-131) both add to the `runner-skill-installation` capability before either is archived** → Use distinct requirement headers so the additive deltas compose cleanly when synced to `openspec/specs/`.
- **Deleting the bash script breaks the e2e until rewiring lands** → Sequence the migration plan so the e2e rewire + setup-staging removal land in the same change as the deletion; run `project_add_feature_e2e_tests.sh` and `project_setup_asdlc_tests.sh` before finishing.

## Migration Plan

1. Implement the TS `repo/` module (ready-path collection + D7 sync), `validate/repo-br-scan.ts`, and `context/repo-br-scan.ts`; register `repo-br-scan` in the `gate` and `context` registries in `cli/run.ts`.
2. Port the gate tests and add context tests (ready-path collection, the D7 block scenarios via git fixtures, no-op, skill-relative asset paths, assembled-block parity); confirm `npm test` green.
3. Author `packages/installer/_data/skills/overmind-repo-br-scan/` (`SKILL.md` with the inlined repo-scan rule + the context→generate→gate→repair loop; `assets/` template + golden example).
4. Generalize `installer/src/init.ts` to install both skills to both runners; update `init.test.ts`.
5. Generalize `project_setup_first_init_machine.sh` skill staging to both skills; remove staging of the migrated command/rule/helper and add them to the obsolete-cleanup; update `project_setup_asdlc_tests.sh`.
6. Rewire `project_add_feature_e2e.sh` phase 4.1 (`run_repo_br_scan_skill`); update `project_add_feature_e2e_tests.sh` expected output/log.
7. Manual smoke: a feature folder with a seeded `feature_br_summary.md` and a ready git repo runs `context repo-br-scan` (sync + assemble) → enrich `## 13` → `gate repo-br-scan` → `0`.
8. Delete the four replaced bash files; remove the test from `CLAUDE.md`; update README/QUICKRUN and the migration design doc. Confirm no dangling references.

Rollback: revert the change; the deleted bash files and the prior e2e/installer wiring return with it (single-commit/PR boundary).
