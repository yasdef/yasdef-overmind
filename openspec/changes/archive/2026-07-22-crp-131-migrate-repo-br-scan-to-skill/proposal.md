## Why

CRP-129 landed the TypeScript foundation (`asdlc-coordinator` + the `overmind capture/context/gate` CLI) and migrated the first pipeline step (`task-to-br`). CRP-130 corrected runner-skill installation so the packaged skill reaches `.codex/skills/` and `.claude/skills/`. The next step in the agreed migration (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md`) is the repo-scan half of step 4.1 — **"Scan repo and apply task-to-BR update"** — still implemented as `feature_scan_repo_for_br.sh` (orchestrator) + `repo_br_scan_rule.md` (rule) + `check_business_context_filled_from_repo.sh` (gate), with `awk`-based parsing. This change migrates that step to an agent **skill** backed by the existing TypeScript core, reusing the patterns proven by CRP-129/130, and folds in the matching installer/setup wiring so no CRP-130-style follow-up is needed.

This step is richer than `task-to-br`: it has **no user-input capture**, but it does enforce the **D7 repo-sync precondition** (default branch, clean tree, upstream, `git pull --rebase`) over the project's **ready** class repositories before scanning, and it gates only `## 1. Document Meta` and `## 13. Existing-System Context` of `feature_br_summary.md`.

## What Changes

- Add the **`repo-br-scan` step** to the shared `asdlc-coordinator` core:
  - `validate/repo-br-scan.ts` — structural gate with behavior parity to `check_business_context_filled_from_repo.sh` (`## 1. Document Meta` `source_type`/`last_updated` filled and `YYYY-MM-DD`; `## 13. Existing-System Context` present with every field filled, no `[UNFILLED]`), registered in the `gate` dispatch registry.
  - `context/repo-br-scan.ts` — deterministic context assembly with parity to `feature_scan_repo_for_br.sh`'s prompt context: resolve the feature path and target `feature_br_summary.md`, resolve `init_progress_definition.yaml` from the nearest ancestor, collect **ready** class repo paths, **sync each ready repo to its default branch (D7)** before scan, and emit one assembled context block (workspace root, target artifact, repos-to-scan list, skill-relative rule/asset references, gate command). No ready classes ⇒ a no-op context block. Registered in the `context` dispatch registry.
  - A shared **`repo/` module** porting `class_repo_paths.sh` (ready-path collection from `meta_info.class_repo_paths`) and `sync_repo_to_default_branch.sh` (D7 default-branch resolution + clean/upstream/rebase enforcement, with rebase-abort on failure) into TypeScript, used by the context builder. The legacy bash `common_libs/*` stay in place — they are still consumed by un-migrated bash steps.
  - No `capture` subcommand for this step: it operates on an already-present `feature_br_summary.md` plus the ready repositories.
- Add the **`overmind-repo-br-scan` skill** (`packages/installer/_data/skills/overmind-repo-br-scan/SKILL.md` with `repo_br_scan_rule.md` inlined; `assets/` BR-summary template + golden example). The model-owned loop: run `overmind context repo-br-scan <feature-path>` → if it reports a no-op, finish → otherwise enrich only `## 1. Document Meta` + `## 13. Existing-System Context` from repository evidence → run `overmind gate repo-br-scan <feature-path>` → finish on `0`, repair-and-rerun on `1`, stop-and-ask on `2`.
- **Fold in installer + setup wiring (so no follow-up is needed):**
  - Generalize `packages/installer/src/init.ts` from a single skill to **all packaged skills** (`overmind-task-to-br`, `overmind-repo-br-scan`), installed into `.codex/skills/` and `.claude/skills/`, with `.overmind/overmind.js` staying the single shared CLI; return metadata for every installed skill path.
  - Generalize the skill-staging in `overmind/scripts/project_mgmt/project_setup_first_init_machine.sh` to stage **both** skills for the supported runners, and **remove** staging of the now-migrated `feature_scan_repo_for_br.sh` command, `repo_br_scan_rule.md` rule, and `check_business_context_filled_from_repo.sh` helper (with update mode cleaning up the stale staged copies).
  - Rewire `overmind/scripts/project_mgmt/project_add_feature_e2e.sh` phase 4.1 to drive the new skill via a Codex session (a `run_repo_br_scan_skill` mirroring the `run_task_to_br_skill` added in commit `1e1305b`) instead of running the deleted bash script: repo scan runs before task-to-BR when a class is ready, and stays a no-op when no class is ready. The Codex launcher stays a thin prompt — model-facing semantic instructions (the final response line, gate exit-code handling, editable surface) live only in each skill's `SKILL.md` as the single source of truth; the launcher never duplicates them. `build_task_to_br_prompt` already follows this single-source rule (its final response line lives only in `overmind-task-to-br/SKILL.md`); `build_repo_br_scan_prompt` mirrors that shape.
- Port the gate + context tests from `tests/ai_scripts/init_scan_repo_for_br_tests.sh` to the TS test runner (including the D7 git-sync block scenarios), and update the `project_setup_asdlc_tests.sh` / `project_add_feature_e2e_tests.sh` shell suites to the migrated wiring.
- **BREAKING / clean break:** remove `overmind/scripts/feature_scan_repo_for_br.sh`, `overmind/rules/repo_br_scan_rule.md`, `overmind/scripts/helper/check_business_context_filled_from_repo.sh`, and `tests/ai_scripts/init_scan_repo_for_br_tests.sh`. No dual bash+TS path. The shared `common_libs/class_repo_paths.sh` and `common_libs/sync_repo_to_default_branch.sh` are **retained** (other bash steps still depend on them until their own migrations).

## Capabilities

### New Capabilities
- `repo-br-scan-skill`: the migrated step 4.1 repo-scan as a whole — the `overmind-repo-br-scan` agent skill, its `repo-br-scan` structural gate (business-context completeness), its context builder (ready-path collection + D7 repo sync + assembled block + no-op handling), and the e2e orchestrator wiring that drives it.

### Modified Capabilities
- `runner-skill-installation`: extended from single-skill to multi-skill installation/staging across the supported runners (Codex + Claude), now including `overmind-repo-br-scan`, plus clean-break removal of the migrated bash command/rule/helper from ASDLC setup staging. (Expressed as additive requirements; `openspec/specs/` is still empty, so there is no prior main-spec requirement to rewrite.)
