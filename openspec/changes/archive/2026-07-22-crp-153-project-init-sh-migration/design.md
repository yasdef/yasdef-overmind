## Context

Init steps 1.1 and 2 are the last model-session shell launchers, and they drag along the last helper/common libraries. Six scripts own this surface:

- `init_project_stack_blueprints.sh` — step 1.1. Type A only (no-op otherwise); for each active `backend`/`frontend`/`mobile` class it builds a prompt binding the blueprint rule file, the class-specific template + golden example, the target `project_stack_blueprint_<class>.md`, the `check_project_stack_blueprint_quality.sh <target>` gate command, the `check_cross_class_peer_trigger.sh <definition>` command, and the external-sources status; loads the `project_stack_blueprint` model row from `.setup/models.md`; runs the model once; and asserts each generated blueprint exists.
- `init_common_contract_definition.sh` — step 2. Binds the common-contract rule/template/golden, the target `common_contract_definition.md`, the gate command, and the peer-trigger command. For type A it passes the approved blueprints as **read-only** project context (snapshots them, runs the model, and asserts they are unchanged); for B/C it passes the ready class-repo paths as authoritative evidence. After the model’s gate loop it commits the initialization baseline (definition + `common_contract_definition.md` + blueprints), asserting no unexpected changes.
- `check_project_stack_blueprint_quality.sh`, `check_common_contract_definition_quality.sh` — deterministic quality gates with `0/1/2` exit codes the model repairs against.
- `check_cross_class_peer_trigger.sh` — no-peer behavior helper (already owned in TypeScript by `repo/cross-class-peer-trigger.ts`).
- `class_repo_paths.sh` — common library that reads/validates `meta_info.class_repo_paths` and `meta_info.project_classes` (already covered in TypeScript by `parse/project-definition.ts` and `repo/collect-ready-paths.ts`).

This is Unit C of `design_docs/e2e_orchestrator_migration/06_sh_remove_plan.md`. Overmind has never been installed, so these scripts are behavior *reference*, not a deployed contract. The contracts of record are `overmind/templates/init_progress_definition_TEMPLATE.yaml`, the `0/1/2` gate semantics, the per-class stack-blueprint templates + golden examples, and the common-contract template + golden example. The already-migrated coordinator is the baseline this change extends and reuses: the generic executor (`runner/execute-step.ts`) and step catalog (`sequencing/step-catalog.ts`, where steps `1.1` and `2` already exist with empty `actions`), the skill/executor architecture and packaged-skill fan-out (`packages/installer`), typed runner config (`config/runner-config.ts`), the `overmind gate` registry and `validate/contract-reconciliation.ts`, `repo/cross-class-peer-trigger.ts`, `parse/project-definition.ts`, `repo/collect-ready-paths.ts`, the read-only-guard machinery (`runner/guards.ts`, `ReadOnlyGuard`), and the project git seam added in Unit B (`ProjectGitPort`/`RepoGitProjectInitAdapter`). `packages/asdlc-coordinator` runtime `dependencies` are empty and stay empty.

## Goals / Non-Goals

**Goals:**

- Add `overmind project init --path <project>` that selects the next pending project-init step from `ProgressReport` (reusing `evaluate`/`nextStep`), applies project-type applicability, and dispatches steps 1.1 and 2 through the existing generic executor — no bespoke launcher, no hand-written step branches.
- Make `StepDefinition.actions` non-empty for steps `1.1` (per-class `stack-blueprint` session) and `2` (project-level `common-contract` session), extending the executor for project-scoped and project-scoped-per-class sessions.
- Package `overmind-stack-blueprint` and `overmind-common-contract` skills (prose + template/golden assets only) and add them to the installer fan-out.
- Port `check_project_stack_blueprint_quality.sh` to `validate/stack-blueprint.ts` with stable `0/1/2` exit codes, exposed as `overmind gate stack-blueprint`; reuse `validate/contract-reconciliation.ts` for the common-contract gate via a thin initial-contract adapter/alias that preserves its target-path and exit-code contract; reuse `repo/cross-class-peer-trigger.ts` for no-peer behavior.
- Preserve the contract behaviors: model prompt bindings, per-class template/golden choice, ready-repo vs. blueprint evidence, read-only blueprint guards during common-contract creation, required outputs (`project_stack_blueprint_<class>.md`, `common_contract_definition.md`), and the project-init baseline commit.
- Fold `class_repo_paths.sh` into `parse/project-definition.ts`, `repo/collect-ready-paths.ts`, and `repo/attach.ts`; load model rows through `config/runner-config.ts`.
- Update `overmind run` pending-work guidance, the sequence diagram, and the definition template to name `overmind project init` and the TypeScript gate contract; delete the six shell files and six shell suites in this change; keep the TypeScript suite + typecheck/lint/format/build green as the standalone bar (full `test:shell`/`npm run verify` green is deferred to the coordinated Unit D landing — see Risks).

**Non-Goals:**

- Worker lifecycle (Unit A / CRP-151), project create + reconcile (Unit B / CRP-152), installer cutover and the repository-wide zero-shell assertion (Unit D / CRP-154), and back-compat residue (Unit E / CRP-155).
- Deleting `project_setup_first_init_machine.sh` itself or `tests/ai_scripts/project_setup_asdlc_tests.sh` (Unit D owns both); this change removes only the `.commands`/`.helper`/`common_libs` staging of *its* six scripts from the first-init machine.
- Porting the shell suites scenario-for-scenario; TypeScript tests specify correct behavior against the artifact contracts, consulting the suites only for genuine edge cases.
- Any new flags/options beyond `overmind project init --path <project>` and `overmind gate stack-blueprint <path>`. Model selection stays in `.setup/models.md` (pipe-table format, loaded through `config/runner-config.ts`); no new runner-config format.
- Changing the model-owned quality loop into coordinator code — the model invokes and repairs against the gates; the coordinator binds gate commands but never runs the loop (architecture invariant 2).

## Decisions

### 1. Reuse the generic executor; steps 1.1 and 2 become catalog session actions

Steps `1.1` and `2` already exist in `STEP_CATALOG` with empty `actions`. This change fills them in rather than adding a launcher: step 1.1 gets a per-class `session("stack-blueprint", "project_stack_blueprint", ["project_stack_blueprint_<class>.md"])` and step 2 gets `session("common-contract", "common_contract_definition", ["common_contract_definition.md"], { readOnlyGuards: [{ mode: "fromContext" }] })`. `modelPhase` reuses the existing `.setup/models.md` rows (`project_stack_blueprint`, `common_contract_definition`) loaded through `config/runner-config.ts`. This satisfies invariant 2 (no bespoke launcher; new sessions are catalog entries executed by the generic executor). *Alternative:* a dedicated project-init executor mirroring the shell — rejected; it duplicates the runner and violates the "no bespoke launcher" invariant.

### 2. `overmind project init` selects the next pending step from `ProgressReport`

`runProjectInit` resolves the runtime root and project path, runs `evaluate(...)`/`nextStep(...)` (the same source `pending-work.ts` uses), and dispatches only the next pending project-init step (`1` is metadata-only with no actions; `1.1` and `2` carry the new sessions). There are no hand-written per-step branches: the verb maps the selected `StepDefinition` to executor bindings and calls `executeStep`. Project-type applicability is applied here — step 1.1 is a no-op for non-type-A projects and for projects with no active `backend`/`frontend`/`mobile` class (matching the shell’s early no-op exits), so `nextStep` advances past 1.1 to 2 in those cases. *Alternative:* a `--step` option — rejected as unrequested surface; next-pending selection matches how `overmind run` already drives steps.

### 3. Project-scoped and project-scoped-per-class executor bindings

Feature sessions bind a `featurePath`; init sessions bind a project path and (for step 1.1) a single class. The executor already supports per-class dispatch (`targetClass`, `<class>` token resolution in outputs/guards) and project-level class-list context (`classListContext`, used by `contract-reconciliation`). This change adds project-scoped session context builders — `buildStackBlueprintContext(projectPath, klass)` (per applicable class) and `buildCommonContractInitContext(projectPath, classes)` — registered in the executor `context`/`classListContext` maps and keyed by the new skill names. Step 1.1 is dispatched once per applicable class by the project-init verb (as the shell looped classes); each dispatch resolves `project_stack_blueprint_<class>.md` for outputs. *Alternative:* a `runIf: "hasReadyClassRepo"`-style predicate for type/class applicability — rejected; blueprint applicability is project-type + class-membership, not ready-repo presence, so it belongs in the verb’s next-step/applicability logic, keeping the predicate set unchanged.

### 4. `validate/stack-blueprint.ts` as a deterministic `0/1/2` gate, registered in `overmind gate`

`check_project_stack_blueprint_quality.sh` becomes `validate/stack-blueprint.ts` returning the standard `GateResult` (`exitCode` `0` pass / `1` business-context fail with `problems` / `2` cannot-run), exported through `validate/index.ts` and registered under `"stack-blueprint"` in the `gateRegistry` in `cli/run.ts` so `overmind gate stack-blueprint <path>` renders the existing pass/`missing:`/`ERROR:` output. Blueprint validation is single-artifact (not per-class at gate time — the model runs it per target path), so it uses the plain `gateRegistry`, not `classGateRegistry`. *Alternative:* a per-class gate keyed by `--class` — rejected; the gate validates one target file the model names, matching the shell’s `<target>` argument.

### 5. Reuse `validate/contract-reconciliation.ts` for the common-contract gate via an initial-contract adapter

`check_common_contract_definition_quality.sh` and the reconciliation gate validate the same `common_contract_definition.md` shape; the design doc directs reuse rather than a second validator. This change adds only the initial-contract adapter/alias needed to preserve the shell gate’s target-path and exit-code contract for *initial* generation (the reconciliation validator is feature/class-list shaped; the adapter exposes the same validation against the project-level target path with `0/1/2` semantics). The `overmind-common-contract` skill’s prose names this gate command; `repo/cross-class-peer-trigger.ts` supplies the no-peer behavior the shell delegated to `check_cross_class_peer_trigger.sh`. *Alternative:* a fresh `validate/common-contract-init.ts` — rejected; it duplicates reconciliation validation and diverges the two contracts.

### 6. Skills carry prose + assets only; the model owns the gate loop

Each skill ships `SKILL.md` (concise normative instructions) plus assets: `overmind-stack-blueprint` carries the per-class templates + golden examples; `overmind-common-contract` carries the common-contract template + golden example. Parsing, mutation, and the gates are TypeScript (invariant 5). The executor binds the gate command, required outputs, and read-only guards; the model invokes `overmind gate stack-blueprint` / the common-contract gate and repairs until `0` (invariant 2). Prompt bindings (rule/template/golden references, target paths, gate + peer-trigger commands, external-sources status, ready-repo vs. blueprint evidence) move into the context builders and skill prose. Prompt-capture tests assert the bindings; executor tests assert the model — not the coordinator — runs the gate loop.

### 7. Read-only blueprint guards via `fromContext`, and the baseline commit as a coordinator post-step

Step 2 must not modify the approved blueprints. The common-contract context builder lists the applicable `project_stack_blueprint_<class>.md` files as `readOnlyInputs` (type A only; empty for B/C), and the step-2 action uses `readOnlyGuards: [{ mode: "fromContext" }]`, so the existing snapshot/verify machinery guards exactly those files — reusing the pattern `contract-delta` already uses, with no new guard mode. The project-init baseline commit is a coordinator concern, not skill prose (invariant 3): after step 2’s gate passes, `runProjectInit` stages the definition + `common_contract_definition.md` + applicable blueprints, asserts no unexpected changes (mirroring `commit_initialization_baseline`), and commits through the Unit-B project git seam (`ProjectGitPort`), keeping project/runtime-root/class-repo git scopes distinct (invariant 4). *Alternative:* committing inside the skill — rejected; runtime mutation must live in typed coordinator code.

### 8. Fold `class_repo_paths.sh` into existing parse/repo modules

`class_repo_paths.sh`’s three jobs already exist in TypeScript: `meta_info.class_repo_paths` reading/validation is `parse/project-definition.ts` `readProjectDefinitionMetadata` (+ `applyClassAttachment` in `repo/attach.ts`), ready-with-existing-directory resolution is `repo/collect-ready-paths.ts` `collectReadyRepoPaths`, and `project_classes` extraction is the same metadata parse. Step 2’s ready-repo evidence and step 1.1’s active-class set are derived from these; no new module is introduced. *Alternative:* a standalone `parse/class-repo-paths.ts` mirroring the shell lib — rejected; it duplicates existing parse/repo surfaces.

## Risks / Trade-offs

- **Project-scoped bindings leak feature-shaped assumptions in the executor** → add explicit project-scoped context builders and reuse the existing `classListContext`/`<class>` seams rather than overloading `featurePath` semantics; cover step 1.1 (per class) and step 2 (project) dispatch with executor tests.
- **The initial-contract adapter diverges from the reconciliation gate over time** → implement it as a thin alias over `validate/contract-reconciliation.ts` (same validation, project target path) rather than a copy, and add a test asserting both accept/reject the same `common_contract_definition.md` shapes.
- **Read-only blueprint guard misses a dynamically-named blueprint** → drive the guard from the context builder’s `readOnlyInputs` (the applicable `project_stack_blueprint_<class>.md` set) via `fromContext`, and add a fixture test that a step-2 run mutating a blueprint fails the guard.
- **Baseline commit stages unexpected paths or races the model’s writes** → port the shell’s exact pathspec set (definition + `common_contract_definition.md` + applicable blueprints) and its "no unexpected changes" assertion, and drive the commit through the injected project git port so tests run without a real repo.
- **Coordinated landing with Unit D (verification bar)** → both `project_setup_first_init_machine.sh` (staging) and `project_setup_asdlc_tests.sh` are Unit-D-owned. The latter copies and asserts the six deleted init assets and the old quickrun shell guidance, and `npm test`/`npm run verify` iterate `tests/ai_scripts/*.sh` via `test:shell` — so `test:shell` (already red on this branch after Unit B) stays red until Unit D removes the wiring and that suite. This change therefore takes the TypeScript suite + typecheck/lint/format/build as its standalone green bar, removes only the staging of its six scripts, grep-checks for residual references, and defers the full-suite/zero-shell green to the coordinated Unit D landing — it does not fix that suite.
- **Pending-work guidance and the sequence diagram/template drift from the shipped verbs** → update `orchestrator/pending-work.ts`, `init_progress_definition_sequence_diagram.md`, and `init_progress_definition_TEMPLATE.yaml` in the same change, and update the `pending-work` tests that currently assert the shell command strings.
