## Why

CRP-141 completed migration of step 8.3 (implementation plan), leaving **step 8.4 â€” "Implementation plan semantic review" (optional)** â†’ `overmind-plan-semantic-review` as the last row in the migration table (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md` Stepsâ†’Skills). It is still implemented as bash: `feature_implementation_plan_semantic_review.sh` (model orchestrator) + `implementation_plan_semantic_review_rule.md` (rule) + `check_implementation_plan_semantic_review_quality.sh` (gate helper). This change migrates that step to the agent-skill + TypeScript-core pattern proven by CRP-129â†’141, completing the feature pipeline migration.

Step 8.4 is an **optional** post-plan pass. It reviews the ordered plan in `implementation_plan.md` for step-level cohesion and split quality, records numbered findings in a **review-ledger** artifact `implementation_plan_semantic_review.md`, asks the operator which findings to apply, and then updates `implementation_plan.md` accordingly. It is therefore the only migrated feature step that **mutates two artifacts** â€” the review ledger it owns and the plan it patches â€” while its structural gate validates only the review ledger.

Unlike step 8.3 (CRP-141), step 8.4 also reads the applicable repo **surface maps** (`project_surface_struct_resp_map_{backend,frontend,mobile}.md`) as read-only inputs (required for each active repo class), because the review's operator-reachability and in-flight-sibling-overlap heuristics inspect them. Like steps 8/8.1/8.2/8.3, it runs once and covers all active repo classes in a single pass; it does not loop per class. Like step 8.3, it has **no pre-session repo sync** and **no capture module** (all upstream inputs are in place, and operator input is collected live in the model session, not written to a capture artifact). It therefore mirrors CRP-139/CRP-141's structure â€” a **context** module + **validate** module + skill â€” with no `sync/` and no `capture/` module.

The gate (`check_implementation_plan_semantic_review_quality.sh`) is a structural review-ledger validator, much smaller than step 8.3's plan gate. It checks that the review artifact has sections `## 1. Document Meta`, `## 2. Review Guidance`, `## 3. Findings Ledger`; that all eight required meta keys are present and filled; that `review_status âˆˆ {in_progress, complete}`; that when no `### Finding N` blocks exist the ledger declares `- no_findings: true` with `review_status: complete`; that `no_findings` is not `true` when Finding blocks exist; and per-finding that all twelve required fields are present and filled, `severity âˆˆ {High, Medium, Low}`, `finding_type` is one of the six allowed types, `state âˆˆ {added, applied, rejected, postponed}`, terminal `delivered_surface_consumption_unclear`/`repo_scaffold_readiness_unclear` findings carry non-empty `resolution_notes`, `delivered_surface_consumption_unclear` references at least one `REQ-*`/`NFR-*` id, and a `complete` review has no non-terminal findings. It also rejects any `[UNFILLED]` placeholder. All of this ports one-for-one to `validate/plan-semantic-review.ts`.

Because CRP-141 already rewired step 8.4's cross-step reference to the plan gate onto the shared CLI (`node .overmind/overmind.js gate implementation-plan <feature-path>`), and step 8.4 is the last row in the table, this migration has **no downstream bash consumer to rewire** â€” a clean-break deletion of the whole step-8.4 bash surface.
