## Why

CRP-129 landed the TypeScript foundation (`asdlc-coordinator` + the `overmind capture/context/gate` CLI) and migrated `task-to-br`; CRP-130 corrected runner-skill installation; CRP-131 migrated `repo-br-scan` and added the shared repo machinery (`collectReadyRepoPaths`, `checkRepoBranchState`, `syncRepoToDefaultBranch`) plus the `overmind sync` verb; CRP-132 migrated step 4.2; CRP-133 migrated step 5; CRP-134 migrated step 5.1; CRP-135 migrated step 6 (contract delta) as `overmind-contract-delta`, adding the per-step `sync` verb, the read-only-input manifest, the ported `repo/list-committed-sibling-features.ts` and `repo/cross-class-peer-trigger.ts`, and the every-exit-path read-only guard. The next row in the agreed migration (`design_docs/to_skills_migration/migration_to_skill_architecture_overview.md` Steps→Skills table) is **step 7 — "Surface map BE/FE"** → `overmind-surface-map` — still implemented as bash: `feature_repo_surface_and_exec_context.sh` (model orchestrator) + `feature_repo_surface_and_exec_context_rule.md` (shared rule) + `check_feature_repo_surface_and_exec_context_be_quality.sh` / `check_feature_repo_surface_and_exec_context_fe_quality.sh` (two per-class gate helpers). This change migrates that step to the agent-skill + TypeScript-core pattern proven by CRP-129→135, and folds in the matching installer/setup/e2e wiring so no follow-up correction is needed.

Step 7 is the first **per-class** step in the migration. For each active project class (`backend`, `frontend`, `mobile`) it produces a separate surface map artifact (`project_surface_struct_resp_map_<class>.md`) by scanning that class's **ready** repository (or, when no repo is ready, falling back to that class's stack blueprint as planned structural evidence). It selects a class to analyze, syncs that class's ready repo to its default branch, binds a per-class template + golden example + quality gate (backend → `_be_` assets/helper; frontend and mobile → `_fe_` assets/helper), and validates the per-class output. Read-only inputs are `init_progress_definition.yaml`, `requirements_ears.md`, `feature_contract_delta.md` (always), plus optionally the class's `project_stack_blueprint_<class>.md` and every committed sibling's `implementation_plan.md` (in-flight promise evidence). There is **no capture** and **no readiness verb**. Per the settled design decision **"Type & per-class branching → resolved by the orchestrator, not SKILL.md conditionals"**, class selection stays in the orchestrator (the existing phase-7 class loop) and the skill/gate/context/sync are **single-class, parameterized by `--class`**.

## What Changes

- Add the **`surface-map` step** to the shared `asdlc-coordinator` core, parameterized by a `--class <backend|frontend|mobile>` option on every verb:
  - `validate/surface-map.ts` — one structural validator with two per-class configs, behavior parity with `check_feature_repo_surface_and_exec_context_be_quality.sh` and `..._fe_quality.sh`. Shared engine: fail (exit `1`) on empty/whitespace target or any `[UNFILLED]`/`[OPTIONAL…]` placeholder; require the per-class title, the four sections (`## 1. Document Meta`, `## 2. Feature Scope`, `## 3. Key Parts of Repo and Their Responsibilities`, the per-class `## 4. … Surfaces Touched With Current Feature`); require nine filled meta keys (`repo_name`, `service_name`, `project_type_code`, `project_classes`, `feature_id`, `feature_title`, `analyzed_repo_paths`, `source_inputs_used`, `last_updated`) with `project_type_code` ∈ {A,B,C} and `last_updated` matching `YYYY-MM-DD`; require three filled scope keys (`feature_summary`, `in_scope_feature_delta`, `out_of_scope_notes`); require the seven per-class layer subsections (`### 3.1…`–`### 3.7…`) each with filled `responsibility_summary`/`main_repo_paths`/`key_components`/`transport_layer`/`user_reachable_surface`, plus the `### 3.8 Another Layer(s)` subsection; require the eight per-class surface subsections (`### 4.1…`–`### 4.8…`) each with filled `surface_summary`/`applicability`/`repo_paths`/`why_feature_touches_it`/`expected_changes`/`evidence`/`transport_layer`/`user_reachable_surface`; require ≥1 surface marked `applicability: applicable`. Per-class config: backend asserts `project_classes` includes `backend` and uses the backend title/section-4 heading/layer+surface names; frontend/mobile assert `project_classes` includes `frontend` or `mobile` and use the frontend/mobile title/section-4 heading/layer+surface names (mobile reuses the frontend config). Exit `0` on pass with the per-class pass message, `1` with the same `quality gate failed: …` messages, `2` on runtime failure. Registered in the `gate` dispatch registry.
  - `context/surface-map.ts` — deterministic context assembly with parity to `feature_repo_surface_and_exec_context.sh`'s `build_prompt`, for the requested `--class`: resolve the feature path + `projects/<id>` root (exit `2` if not under `projects/<id>/<feature>`); resolve read-only `init_progress_definition.yaml`, `requirements_ears.md`, `feature_contract_delta.md` (exit `2` if any absent); verify the requested class is an active `meta_info.project_classes` member (exit `2` otherwise); resolve the class's **ready** repo path via the shared `collectReadyRepoPaths` (running `checkRepoBranchState`, block/exit `2` verbatim) **or**, when no repo is ready, fall back to `project_stack_blueprint_<class>.md` as planned structural evidence (exit `2` if neither a ready repo nor a blueprint exists); collect committed siblings' `implementation_plan.md` (via the reused `listCommittedSiblingFeatures`) as in-flight read-only evidence; emit one context block with the per-class binding (track label, template/golden asset refs, target artifact `project_surface_struct_resp_map_<class>.md`, allowed-write list = that one file), the scan scope (ready repo path or blueprint-fallback note), one `- read_only_input: <workspace-relative-path>` manifest entry per resolved read-only input (the three always-on inputs + the optional blueprint + the in-flight plans), the `project_classes` meta value for the run, and the exact `surface-map --class <class>` gate command. Registered in the `context` dispatch registry.
  - `sync/surface-map.ts` — `overmind sync surface-map <feature-path> --class <class>` syncs the requested class's **ready** repo to its default branch before the session (reusing `collectReadyRepoPaths` + `syncRepoToDefaultBranch`); a blueprint-fallback class (no ready repo) is a no-op. Registered in the `sync` dispatch registry.
- Add the **`overmind-surface-map` skill** (`packages/installer/_data/skills/overmind-surface-map/SKILL.md` with `feature_repo_surface_and_exec_context_rule.md` inlined; `assets/` with **both** the `_be_` and `_fe_` templates + golden examples). The model-owned loop: run `overmind context surface-map <feature-path> --class <class>` → read the per-class binding + scan scope from the context block → scan only the listed repo path (or rely on the blueprint fallback as primary planned structural evidence) → draft `project_surface_struct_resp_map_<class>.md` from the bound template/golden → write only that one file, never modifying any read-only input → run `overmind gate surface-map <feature-path> --class <class>` after every write, repair on `1`, stop-and-ask on `2`, finish on `0`. The two literal final-response lines (parameterized by track label) live only in `SKILL.md`: the success line `Repo surface and execution context <track> phase is finished. Nothing else to do now; press Ctrl-C so orchestrator can start the next phase` and the infeasibility line `repo surface and execution context <track> gate cannot pass with current repository evidence. Please provide instructions what to do, or adjust requirements and rerun this phase`.
