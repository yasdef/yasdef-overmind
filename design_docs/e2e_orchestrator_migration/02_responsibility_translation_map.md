# E2E Orchestrator Migration — 2. Responsibility Translation Map (sh → TS)

How each responsibility group from `01_current_e2e_functional_inventory.md` lands in TypeScript. **Existing** = already implemented in `packages/asdlc-coordinator` on this branch; **New** = created by this migration. Nothing ports 1:1 — bash text-parsing seams (stdout scraping, awk YAML slicing, numeric rc protocols) are replaced by typed in-process calls; only genuinely new capability is written from scratch.

Guiding rule (unchanged from the skills migration): TS owns deterministic mechanics; skills own model-facing semantics; the model owns the gate loop. The orchestrator may **import** validator/readiness/context functions in-process, but must never run the `overmind gate` CLI verb itself (see `design_docs/to_skills_migration/step_by_step_migration_for_particular_step.md ## 2. Ownership Rules`).

## Deleted with no successor (dissolved by the architecture)

- Bash utility layer: `trim_value`, `to_lower`, `array_contains`, `compare_dotted_steps`, `is_dotted_numeric_step`, all awk YAML slicing — native TS / existing parsers.
- Scanner stdout contract (`next step:` line parsing) — producer and consumer share types.
- `map_scanner_step_to_phase` remapping tables (legacy `4` → `5`, name-substring fallbacks) — they compensated for the old scanner's imprecise step reporting; with the definition-as-spec scanner, catalog ids equal definition ids and the layer has nothing left to translate.
- `MODEL_CMD=codex` string assertions scattered per phase — one config-validation rule.
- The 13-fold copy-paste of the launcher pattern — one data-driven executor.
