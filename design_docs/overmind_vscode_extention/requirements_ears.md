# Requirements (EARS) - Overmind VS Code Extension

System name: Overmind VS Code Extension
Scope: Local VS Code extension that provides BA/PO operators with a dashboard and guided controls for an ASDLC workspace. The ASDLC filesystem remains the source of truth.

---

## Overview
- Product/Domain: Operator-facing UI for Overmind ASDLC workspaces.
- Goals: Show projects, features, readiness, repo/class status, and artifact access without requiring operators to type CLI commands.
- Out of scope: Duplicating coordinator behavior, storing authoritative workflow state outside ASDLC files, cloud synchronization, multi-user collaboration.

## Glossary
- ASDLC workspace: Local folder generated by Overmind setup, containing `asdlc_metadata.yaml`, `.overmind/`, and `projects/`.
- Project: ASDLC project folder under `projects/<project-id>`.
- Feature: Feature planning folder under `projects/<project-id>/<feature-folder>`.
- Readiness: Completion status computed from known ASDLC artifacts and the declared workflow steps.
- Artifact: Markdown, YAML, or text file generated by the Overmind workflow.
- Operator: BA/PO user who needs to inspect or drive Overmind without using raw CLI commands.

## Actors
- BA/PO Operator: Uses the extension to understand project and feature planning state.
- Technical Operator: Uses the extension to run allow-listed `overmind` actions with visibility into interactive terminal output.
- Developer/Maintainer: Maintains the coordinator core, workflow assets, and the extension package.

## Assumptions
- Operators have VS Code installed or can install it.
- The ASDLC workspace is available on the local filesystem or through VS Code Remote WSL.
- Shared coordinator primitives and shipped `overmind` verbs remain the authoritative implementation of workflow actions.
- Version 1 is read-only and must not mutate ASDLC files.
- Future guided task-to-BR capture forms will call the shared Overmind core capture primitive instead of duplicating `user_br_input.md` file-writing rules in the Webview. Jira capture records a ticket marker; the task skill/context step remains responsible for MCP fetch and persistence of fetched story text.

## Requirements

### Requirement 1 - ASDLC Workspace Detection
**User Story:** As a BA/PO operator, I want the extension to detect an ASDLC workspace, so that I do not need to configure technical paths manually.

**Acceptance Criteria (EARS):**
- WHEN a VS Code workspace contains `asdlc_metadata.yaml`, THE Overmind VS Code Extension SHALL identify that workspace as an ASDLC workspace.
- WHEN no ASDLC workspace is detected, THE Overmind VS Code Extension SHALL show a clear empty state with instructions to open or select an ASDLC folder.
- IF multiple workspace folders are open and more than one contains `asdlc_metadata.yaml`, THEN THE Overmind VS Code Extension SHALL ask the operator to select the active ASDLC workspace.

**Verification:** Extension integration tests cover single-folder, no-folder, and multi-folder workspace detection.

---

### Requirement 2 - Project Dashboard
**User Story:** As a BA/PO operator, I want to see existing ASDLC projects, so that I can understand active planning work without reading folder structures.

**Acceptance Criteria (EARS):**
- WHEN an ASDLC workspace is detected, THE Overmind VS Code Extension SHALL list projects from `asdlc_metadata.yaml`.
- WHEN a listed project folder is missing, THE Overmind VS Code Extension SHALL mark the project as inconsistent instead of failing the dashboard.
- THE Overmind VS Code Extension SHALL show each project name, project id, creation date when available, project type, active repo classes, and class repo readiness.

**Verification:** Coordinator and extension unit tests validate project extraction from representative `asdlc_metadata.yaml` and `init_progress_definition.yaml` fixtures.

---

### Requirement 3 - Feature Dashboard
**User Story:** As a BA/PO operator, I want to see features inside each project, so that I can track planning readiness feature-by-feature.

**Acceptance Criteria (EARS):**
- WHEN a project folder contains feature folders, THE Overmind VS Code Extension SHALL list those features under the project.
- THE Overmind VS Code Extension SHALL show each feature name, detected step completion, missing required artifacts, and readiness summary.
- IF a folder under a project is not a feature folder, THEN THE Overmind VS Code Extension SHALL ignore it unless it contains `feature_br_summary.md`.

**Verification:** Coordinator and extension unit tests cover complete features, incomplete features, and non-feature folders.

---

### Requirement 4 - Readiness Computation
**User Story:** As a BA/PO operator, I want a simple readiness signal, so that I can quickly know whether a feature is ready for implementation.

**Acceptance Criteria (EARS):**
- THE Overmind VS Code Extension SHALL compute total project readiness from project-level initialization steps and feature-level readiness.
- THE Overmind VS Code Extension SHALL compute feature readiness from expected Overmind artifacts and available checklist state.
- THE Overmind VS Code Extension SHALL compute repo/class readiness from `meta_info.project_classes` and `meta_info.class_repo_paths` in `init_progress_definition.yaml`.
- IF readiness cannot be computed confidently, THEN THE Overmind VS Code Extension SHALL show an `unknown` state with the coordinator diagnostic.

**Verification:** Readiness unit tests validate complete, partial, missing, deferred, and unknown states.

---

### Requirement 5 - Artifact Access
**User Story:** As a BA/PO operator, I want to open generated artifacts from the dashboard, so that I can review requirements and plans without browsing folders manually.

**Acceptance Criteria (EARS):**
- WHEN an artifact exists, THE Overmind VS Code Extension SHALL provide an action to open it in VS Code.
- WHEN an expected artifact is missing, THE Overmind VS Code Extension SHALL show it as missing and SHALL NOT provide a broken open action.
- THE Overmind VS Code Extension SHALL group artifacts by project-level and feature-level scope.

**Verification:** Extension integration tests verify file-open actions for existing artifacts and disabled actions for missing artifacts.

---

### Requirement 6 - Progress Refresh
**User Story:** As a BA/PO operator, I want to refresh dashboard data, so that I can see the current ASDLC filesystem state after actions or files change.

**Acceptance Criteria (EARS):**
- WHEN the operator clicks refresh, THE Overmind VS Code Extension SHALL recompute progress in process through `asdlc-coordinator/sequencing`.
- WHEN relevant ASDLC files change on disk, THE Overmind VS Code Extension SHALL trigger a fresh in-process progress computation.
- IF computation reports a file-level diagnostic, THEN THE Overmind VS Code Extension SHALL report the affected path and render the degraded result.

**Verification:** Coordinator and extension tests cover manual recomputation, file watcher recomputation, and diagnostic rendering.

---

### Requirement 7 - Allow-Listed Overmind Actions
**User Story:** As a technical operator, I want dashboard buttons to invoke supported Overmind actions, so that I can run workflow steps without typing commands.

**Acceptance Criteria (EARS):**
- WHERE actions are enabled, THE Overmind VS Code Extension SHALL invoke only allow-listed shipped verbs: read-only `overmind status` and terminal-hosted `overmind run`. Feature creation SHALL use the in-process `scaffoldFeature()` coordinator primitive, not a shipped verb.
- WHEN `overmind run` requires interactive model sessions, THE Overmind VS Code Extension SHALL host it in a visible VS Code integrated terminal.
- BEFORE invoking a mutating action, THE Overmind VS Code Extension SHALL show the exact project or feature path and ask for confirmation.
- IF neither the workspace `.overmind/overmind.js` nor the bundled coordinator core is available, THEN THE Overmind VS Code Extension SHALL show a blocking action error.
- WHERE project creation is requested, THE Overmind VS Code Extension SHALL postpone that action until a shared coordinator primitive or shipped `overmind` verb exists.

**Verification:** Integration tests validate command construction and terminal launch behavior using mocked VS Code terminal APIs.

---

### Requirement 8 - Guided Webview Forms
**User Story:** As a BA/PO operator, I want simple forms for common Overmind actions, so that I can create projects and features without answering shell prompts.

**Acceptance Criteria (EARS):**
- WHERE guided forms are implemented, THE Overmind VS Code Extension SHALL collect required inputs in Webview forms before invoking an Overmind action.
- THE Overmind VS Code Extension SHALL validate form inputs before any filesystem mutation is attempted.
- IF an action runs model sessions, THEN THE Overmind VS Code Extension SHALL use terminal-hosted `overmind run`; deterministic actions SHALL use coordinator capture/scaffold primitives and `InteractionPort`-backed forms.

**Verification:** Webview tests cover required fields, invalid paths, cancellation, and valid action payloads.

---

### Requirement 9 - Preserve ASDLC Source of Truth
**User Story:** As a maintainer, I want the extension to avoid duplicating Overmind workflow rules, so that behavior stays consistent with the coordinator core.

**Acceptance Criteria (EARS):**
- THE Overmind VS Code Extension SHALL treat ASDLC files and shared coordinator primitives as the source of truth.
- THE Overmind VS Code Extension SHALL NOT persist authoritative project, feature, or readiness state outside the ASDLC workspace.
- WHEN the extension needs durable configuration, THE Overmind VS Code Extension SHALL store only extension preferences, not workflow state.

**Verification:** Code review confirms workflow mutation paths call coordinator capture/scaffold/sync primitives or shipped `overmind` verbs.

---

### Requirement 10 - Task-To-BR Input Capture Form
**User Story:** As a BA/PO operator, I want to capture story input for task-to-BR through a guided UI, so that I do not need to manually create `user_br_input.md`.

**Acceptance Criteria (EARS):**
- WHEN the operator starts task-to-BR capture for a feature, THE Overmind VS Code Extension SHALL let the operator choose exactly one source: a local `.txt`/`.md` story file inside the feature folder or a Jira ticket identifier.
- WHEN the operator submits a local story file or Jira ticket, THE Overmind VS Code Extension SHALL call the shared Overmind core capture primitive for `task-to-br` and SHALL NOT duplicate `user_br_input.md` rendering logic in Webview code.
- WHEN the operator submits a Jira ticket, THE Overmind VS Code Extension SHALL treat capture as ticket-marker persistence and SHALL leave Jira story fetch/persistence to the shared task-to-BR skill/context flow.
- WHEN capture succeeds, THE Overmind VS Code Extension SHALL refresh the feature dashboard and show `user_br_input.md` as present.
- IF capture fails validation, THEN THE Overmind VS Code Extension SHALL show the core error without mutating unrelated artifacts.

**Verification:** Extension action tests verify form validation and mocked calls into the shared core capture API for local-file and Jira capture.

## Non-Functional Requirements

### NFR 1 - Cross-Platform Operation
**User Story:** As an operator, I want the dashboard to work on Mac, Linux, and Windows WSL2, so that all BA/PO environments are supported.

**Acceptance Criteria (EARS):**
- THE Overmind VS Code Extension SHALL use VS Code workspace and filesystem APIs where practical to support local and Remote WSL workspaces.
- WHERE terminal-hosted `overmind run` is used on Windows, THE Overmind VS Code Extension SHALL support the active VS Code Remote WSL execution context.

**Verification:** Manual smoke tests cover macOS, Linux, and Windows WSL2.

---

### NFR 2 - Safety
**User Story:** As an operator, I want safe workflow actions, so that accidental mutations are unlikely.

**Acceptance Criteria (EARS):**
- THE Overmind VS Code Extension SHALL keep read-only dashboard features separate from mutating workflow actions.
- BEFORE any mutating action, THE Overmind VS Code Extension SHALL require explicit user confirmation.
- THE Overmind VS Code Extension SHALL NOT run destructive git or filesystem commands directly.

**Verification:** Action tests verify confirmation gates and `overmind` verb allow-listing.

---

### NFR 3 - Performance
**User Story:** As an operator, I want the dashboard to load quickly, so that checking project state is low-friction.

**Acceptance Criteria (EARS):**
- THE Overmind VS Code Extension SHALL render the initial dashboard within 2 seconds for a typical ASDLC workspace containing up to 50 projects and 500 features on a local SSD.
- THE Overmind VS Code Extension SHALL perform progress computations asynchronously so the VS Code UI remains responsive.

**Verification:** Performance smoke test runs against generated fixture workspaces.

---

### NFR 4 - Diagnostics
**User Story:** As a maintainer, I want actionable diagnostics, so that broken ASDLC workspaces can be fixed quickly.

**Acceptance Criteria (EARS):**
- THE Overmind VS Code Extension SHALL report parse errors with file path and reason.
- THE Overmind VS Code Extension SHALL expose a diagnostic output channel for progress computation and action events.
- THE Overmind VS Code Extension SHALL avoid exposing sensitive file contents in routine diagnostics.

**Verification:** Diagnostic tests cover YAML parse errors, unavailable coordinator/CLI surfaces, missing project folders, and permission failures.
